# 目標管理 UI/UX 再考 — ブレインストーミング

> 目的: 現状の「目的 + ペース」型の目標管理を起点に、UI/UX を発散的に再考する。最終決定ではなく**選択肢を広げる**ためのドキュメント。
> 起点となる課題感: **進捗・モチベーションが見えにくい**
> 関連 PRD: [PRD §6.3 / §6.6 / §6.7](../PRD.md)

---

## 0. 現状サマリー (出発点の整理)

### 0.1 いまの目標管理の構造
- **定義要素**: `goalDirection` (lose/maintain/gain) × `paceLevel` (gentle/standard/strong)
- **派生値**: `targetWeightKg` / `targetBodyFatPct` / `targetCalories` / `P/F/C g`
- **時間軸**: 内部的には「3ヶ月後の予測」だが UI では非表示 (`computePlanOutcome` は 13週前提)
- **編集導線**: My Status → 目標カード → `/goal-edit` (segmented control 2本 + プレビューカード)
- **進捗表示**: My Status の Hero に「現在体重 / 目標体重」「現在BF% / 目標BF%」を併記。差分や 7日平均は出るが「目標までの距離」「達成予定日」は無い

### 0.2 現状の体感的な不足 (仮説)
1. **数字だけで距離感が伝わらない** — `60.2 kg → 58 kg` は引き算しないと残量が見えない
2. **時間軸が見えない** — 「3ヶ月後」の前提があるのに UI ではその文脈が消えている
3. **積み上げの誇り (streak / 累積) がない** — 食事ログは別軸、体重記録はカードに埋もれる
4. **目標体型 (見た目) と数値目標が別の場所にある** — profile の 9マトリクスに行かないと「ありたい姿」が見えない
5. **編集と確認が同じ画面に同居していない** — 目標カードは確認、`/goal-edit` は編集、`/stats` は実績、と分散

## 1. 観点A: 目標の定義方法そのもの

「目的 + ペース」だけだと抽象的すぎる、という前提で別の表現を発散する。

### A-1. 「期日」を一級市民にする
- 目標: 「6/30 までに 56.0 kg」
- ペースを直接選ばず、**期日 + 目標値 → ペース逆算** にする
- メリット: カレンダー上で進捗が直感的になる、リマインドや「あと○日」訴求がしやすい
- デメリット: 期日固定はプレッシャーになる、医療リスク (極端設定) のガード強化が必要
- バリエーション:
  - A-1a: **柔らかい期日** (「だいたい夏ごろ」3段階のテンポを内部マップ)
  - A-1b: **マイルストーン分割** (4週ごとに小目標、達成で次のマイルストーン提示)

### A-2. 「達成すると体型がこう変わる」を主役に
- 目的・ペースは裏に隠し、**目標体型シルエット** + **3ヶ月後ビフォーアフタープレビュー** をメインに据える
- 目的 (減/維/増) の選択画面そのものを廃止し、**目標体型タイル選択 → direction 自動算出** に戻す案 (v1.3 で廃止した 9マトリクス選択を「**ありたい姿**」として復活)
- メリット: PRD §1 の「見た目で目標を考える」原則に最も近い、感情的な動機づけ強い
- デメリット: v1.3 で減らしたばかりの認知負荷が戻る、シルエット精度に左右される

### A-3. 「健康レンジ + 微調整」モデル
- 目的・ペースを廃し、**健康レンジ (BMI 20–24 など) のバンド** + **その中での自分のポジション** をスライダーで指定
- 数値ではなく**バンド内の相対位置** が UI のメイン
- メリット: 完璧主義を緩和、「まずレンジ内に入る」のような中継目標が成立
- デメリット: BF% や筋量という第二軸が失われる (PRD §3.1 の 2軸思想と相性が悪い)

### A-4. 「複数目標 (体重 / BF% / 行動)」の併用
- いまの目標 = 体重・BF%・kcal 一括
- これを **3つの独立した目標** に分割: ① 体型 (見た目) ② 数値 (体重/BF%) ③ 行動 (週○回ログ・週○回運動)
- メリット: モチベーション源を多様化、「数値が動かなくてもログ習慣が続いている」と肯定できる
- デメリット: 概念が増え、Minimal Hub 原則 (§6.6) と緊張する

### A-5. 「ジャーニー」モデル
- 目標を**到達点ではなくフェーズの連鎖**として定義: 「第1段階: 食事を記録する」→「第2段階: -1kg」→「第3段階: 体型を ◯◯ にする」
- メリット: 早期に達成感、長期目標の心理的ハードル低下
- デメリット: PRD のシンプルさと衝突、設計コスト大

### A-6. 「目標なし」状態の許容
- 「まだ決めていない / 様子見」を一級モードとして用意
- いまは onboarding で必須だが、**デフォルト = 維持(ニュートラル)** で開始 → ログを溜めながら My Status 内で目標誘導
- メリット: 離脱率改善、後から目的が固まるユーザーの実体に合う
- デメリット: KPI (オンボ完了率/目標プラン確定率) の定義変更が必要

## 2. 観点B: goal-edit 画面のレイアウト/操作性

現状: プレビューカード / 目的セグメント / ペースセグメント / 保存ボタン の縦積み。

### B-1. ライブシミュレーター強化
- 上部に**シルエット差分アニメーション** (現在体型 → 目標体型へのモーフィング、3秒ループ)
- 下部に**3ヶ月折れ線プレビュー** (体重推移シミュレーション、dotted line で目標線、shade で誤差レンジ)
- 現状の単一プレビューカードを「**ビフォーアフター**」と「**推移グラフ**」に分割

### B-2. プラン比較カード (オンボの再利用)
- onboarding ではプラン3カード並列比較になっているが、goal-edit はセグメントだけ
- goal-edit にも **3カード並列** (横スワイプ) を持ち込み、選んだ瞬間に保存
- メリット: 認知の一貫性、選択時に他選択肢との対比が見える
- デメリット: 既存セグメントのコンパクトさを失う

### B-3. インラインで全部見せる「ダッシュボード型」
- 1画面の中に**現在体型・目標体型・3ヶ月予測グラフ・kcal/PFC・推奨食事メッセージ** をすべて展開
- スクロール許容
- メリット: 目標管理の全体像が一目で見える
- デメリット: PRD §6.6 の Minimal 原則と衝突する可能性

### B-4. ウィザード型 (3ステップ)
- ① 目的 → ② ペース → ③ 数値確認 (微調整)
- いまは1画面に詰めているが、ステップ化することで**選んだ理由を意識させる**
- メリット: ペース選択時に副作用 (kcal の変化) を強調できる、モチベ醸成可能
- デメリット: クリック数増、頻繁に編集する人には冗長

### B-5. 「いま」を起点に変化を選ぶインタラクション
- 数値 (kcal や体重) のスライダーを直接動かす
- スライダー操作で **direction / pace を逆推定** する
- メリット: 数値ベースで考えたいユーザーに合う、操作の手応えがある
- デメリット: PRD v1.3 で「手動 ± 微調整は廃止」した方針と衝突

### B-6. 「サマリー → タップで深掘り」
- 通常時はサマリーだけ (1画面 / 2タップで完結)
- 「詳しく」を押すと**根拠ロジック (RMR/TDEE/活動係数)** を展開
- メリット: 簡素 + 透明性の両立、上級者の信頼獲得
- デメリット: コンテンツ二重化

## 3. 観点C: 現在体型と目標体型の関係表現

v1.3 で 9マトリクス目標選択を廃止 → direction から自動導出にしたが、視覚的な「現在 → 目標」の橋渡しが弱い。

### C-1. 9マトリクス上で「いまの位置」と「行き先」を矢印表示
- 3×3 のマトリクスを **視覚インデックス** として復活 (選択 UI ではなく**ステータス表示**)
- 現在セルをハイライト + 矢印で目標セルを指す
- 編集はせず、参照専用
- メリット: PRD §3.1 の「同じ概念体系」が UI で復活、見た目の旅路が一目
- デメリット: 9マトリクスを再露出することの是非 (v1.3 廃止の理由と再整合が必要)

### C-2. シルエットの **アニメーション補間**
- 現在体型と目標体型のシルエットを並べ、**スライダーで補間** (0% = 現在, 100% = 目標)
- 「いま 25% の場所にいる」のように進捗を示す
- メリット: 実進捗が見た目で分かる、モチベ強い
- デメリット: 18体しかないので途中体型は補間で擬似生成、不自然になりうる

### C-3. **二点散布図** (体重 × BF%)
- 横軸: 体重、縦軸: BF%
- 現在点 (青) → 目標点 (緑) を結ぶ矢印
- 過去ログを薄く表示 (軌跡)
- メリット: 数値派にウケる、矢印で旅の方向が見える
- デメリット: グラフリテラシーが必要、健康アプリにしては硬い

### C-4. 「体型バッジ」を Hero に常駐
- My Status Hero に**現在体型ラベル**を小さく表示 (例: "脂肪ふつう × 筋量ふつう")
- タップで profile の 9マトリクスへジャンプ
- 目標体型ラベルも同行に併記 → 「いま」と「目標」が常に並んで見える
- メリット: 露出強化が低コスト、PRD 中核思想の常時可視化
- デメリット: PRD §6.6.4 の「体型タイル表示はトップに出さない」と衝突 (再議論要)

### C-5. **プログレスメーター** (体型距離)
- 「現在体型から目標体型への変化を 100% スケールで表現」
- 0% = 開始日 / 100% = 目標達成 / 円グラフや帯グラフで可視化
- 進捗 = (現在の体重と現在体型タイル中央値からの偏差) との関数で動的計算
- メリット: 達成感が時間で増える、ゲーミフィケーション余地あり
- デメリット: 計算根拠の透明性確保が必要 (途中で停滞すると逆効果)

## 4. 観点D: 進捗表示 (My Status Hub での目標見え方) — **メイン課題**

### D-1. 「あと ○○ kg / あと ○○ 日」訴求
- Hero に **「あと 2.2 kg · あと 47 日」** を最も大きく表示
- 体重・BF% より上位の情報として配置
- メリット: 距離感と時間軸の両方が一目
- デメリット: 期日設計が必要 (観点 A-1 と連動)

### D-2. **進捗バー** + マイルストーン
- 「開始 → 現在 → 目標」の横バー
- 25/50/75% 地点にマイルストーン (祝マーク)
- メリット: 累積感、達成バッジ余地
- デメリット: 目標が後ろに後退する (体重リバウンド) 時の表示が難しい

### D-3. **トレンドスパークライン**
- 目標カード内に過去30日の体重スパークラインを内蔵
- 目標値を点線で重ねる
- 7日移動平均線も併記
- メリット: 推移が一目、停滞期の心理的ガードに
- デメリット: 既存 stats 画面との役割重複

### D-4. **デイリーストリーク** をモチベ装置として導入
- ログを連続して付けた日数 ("8日連続")
- 体重記録ストリーク、食事ログストリークを別管理
- メリット: 行動継続のリワード、「数値が動かなくても自分は続いている」効果
- デメリット: ストリーク途切れ時の心理的逆効果 (Duolingo 病)

### D-5. **週レビュー カード** を My Status に常駐
- 毎週月曜の朝に「先週は...」が Hero 下に出る
- 平均kcal / ログ日数 / 体重変化 / コメント
- 「今週の目標 (例: ログを5日)」設定
- メリット: 短期サイクルが回る、ジャーナリング的な感情価値
- デメリット: 通知設計とセット、週末ユーザーには合わない可能性

### D-6. **目標達成度スコア** (0-100)
- 体重達成度 + 行動達成度 を合算した1スコア
- 例: 「目標達成度 62 / 100」
- メリット: 単一指標で語れる、ゲーミフィケーション
- デメリット: スコア化の根拠説明が必要、過度な競争を避ける配慮要

### D-7. 「**変化を祝う**」モーメント設計
- -0.5 kg, -1 kg, -2 kg... のしきい値到達時にトースト/モーダル
- 目標体型の更新シルエット
- メリット: 短期サイクルの感情ピーク
- デメリット: トリガー条件と頻度の調整が難しい

### D-8. **タイムライン形式の目標カード**
- 「3/15 開始 (60.0 kg) → 5/1 現在 (58.5 kg, -1.5 kg) → 6/30 目標 (56.0 kg)」
- 縦タイムラインで過去 / 現在 / 未来を一望
- メリット: 自分の旅路が明示、ストーリー性
- デメリット: スペースを取る

### D-9. **Apple Fitness/Watch 風 リング** の体重達成版
- 円リングが目標達成度に応じて埋まる
- 視覚的にすぐ分かる
- メリット: 既存メンタルモデルに乗る、感覚的
- デメリット: 「リバウンドでリングが減る」体験は微妙

## 5. 組み合わせの発散 (複合案)

ブレストの最後に、**よさげな組み合わせ** をいくつか先回りで束ねておく。これは「次の収束フェーズで叩く素材」として並べておくだけ。

### 組み合わせ案 α: 「期日 + シルエット補間 + マイルストーン」
- A-1 (期日を一級市民) + C-2 (シルエット補間) + D-2 (進捗バー + マイルストーン)
- ストーリー: 「6/30 までに ◯◯ 体型」を選び、いまの位置をシルエット補間で見せ、25/50/75% にマイルストーン
- 強み: 時間軸 / 見た目 / 累積 の三位一体

### 組み合わせ案 β: 「複数目標 + 週レビュー + ストリーク」
- A-4 (体型/数値/行動の3目標) + D-5 (週レビュー) + D-4 (ストリーク)
- 強み: 数値が動かない週もログ習慣で勝てる、行動価値を主軸に置けるダイエット非依存モデル

### 組み合わせ案 γ: 「マトリクス常駐 + ビフォーアフター強化」
- C-1 (マトリクス上の矢印表示) + B-1 (シルエット差分アニメ) + D-1 (あと kg / あと 日)
- 強み: PRD 中核 (9マトリクス) を視覚軸で復活させつつ、編集 UI は v1.3 のまま

### 組み合わせ案 δ: 「ジャーニー + 達成度スコア + 祝う瞬間」
- A-5 (ジャーニー) + D-6 (スコア) + D-7 (達成モーメント)
- 強み: 長期離脱を防ぐ感情的フック、ゲーミフィケーション全振り
- 弱み: PRD トーン (静かな日本語ウェルネス) との緊張

## 6. 議論の足場 (収束フェーズの問い)

ブレストから収束に移る時に拾う論点:

1. **時間軸を一級市民にするか?** (A-1 を採用するなら期日 UI 必須)
2. **9マトリクスをトップに常駐させるか?** (C-1/C-4 が PRD §6.6.4 と再整合できるか)
3. **目標を「数値」中心 / 「見た目」中心 / 「行動」中心 のどれに重みを置くか?** (A-4 を分割で持つか、メイン目標を1つ選ぶか)
4. **ゲーミフィケーション要素の量** (静かなウェルネストーンとの両立)
5. **編集 UI と確認 UI の融合度** (B-3 ダッシュボード型は本当に望ましいか)
6. **進捗が「停滞」「後退」した時の表現** (リバウンド・記録欠落への配慮)

---

## 7. このドキュメントの使い方

1. **発散フェーズの素材集** として読む — 各案は採用前提ではなく、議論のための呼び水
2. 気に入った案 / 即没案 / 保留案を仕分けする
3. 仕分け結果を踏まえ、次のフェーズで **2-3 案の比較プロトタイプ** に絞り込む
4. 絞り込み後、PRD §6.3 / §6.6 を更新 → 実装着手

---

> 次アクション候補:
> - 案を絞った上で、ワイヤフレーム/モックを描く (B/C/D の組み合わせ別に 2-3 案)
> - PRD への反映ドラフトを起こす (採用案が固まった時点で)
> - 検証手段を決める (社内レビュー / プロトタイプ A/B / ユーザーインタビュー)
