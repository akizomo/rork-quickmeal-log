## 1. Problem Statement

### 1.1 現状
MVP 段階の Quick Log は **9 ボタン × Identity ツリー × 自動学習** の3段構造で、9バケットのいずれかに Identity を抽象化して収録している。Identity 全件検索バーはコードとして実装済みだが、UI からは非表示になっている (PRD §4.2)。

### 1.2 解決したい問題
- 9 ボタン経由でも長押し階層展開でも目的 Identity に届かないケースが発生する (例: ブランド指定したい / 特殊な料理名で探したい / コンビニ商品名で直接検索したい)
- このフォールバック手段がないと「あすけんなら載っているのに、ここにはない」という体感を生み、競合 (あすけん主軸) に対する離脱要因になる
- 一方で検索を主導線化すると、9ボタン × 自動学習というコア体験の軽さ (PRD §6.5.0 解釈A) を毀損するリスクがある

### 1.3 解決方針 (PRD v1.6 で確定)
- **解釈A**: 検索バーは「ボタン+長押しで届かなかった時の最後の砦」として再有効化する
- **DB件数で勝負しない**: 1,000〜3,000 Identity + ブランドチップ数百〜千件で十分
- **コア導線は変えない**: 9 ボタンが主導線、検索バーはフォールバック

## 2. User Stories

### 2.1 Primary
- **U1**: ユーザーとして、9 ボタンで思った Identity が見つからないとき、画面上部の検索バーから商品名・料理名・食材名で直接検索したい (例: 「サラダチキン」「カフェラテ」「鶏むね」)
- **U2**: ユーザーとして、ひらがな・カタカナ・ローマ字のどれで入力しても同じ結果が得られたい (例: 「ぶろっこりー」「ブロッコリー」「broccoli」)
- **U3**: ユーザーとして、よく検索する Identity は次回検索時に上位に表示されたい
- **U4**: ユーザーとして、検索結果がない場合、何ができるか (リクエスト送信 / カスタム登録) を提示されたい

### 2.2 Secondary
- **U5**: ユーザーとして、検索結果から選んだ後は通常の Quick Log と同じフロー (量入力 → 確定) で記録できたい
- **U6**: ユーザーとして、検索バーを閉じれば 9 ボタン主導線に戻れたい (検索を主役にしない)

## 3. Functional Requirements

### 3.1 Must Have (Next での確定実装範囲)

#### F1. 検索バー UI 再表示
- Quick Log 画面上部に検索バーを再配置
- プレースホルダー: 「食材・料理・ブランド名で検索」
- 既存実装 (`expo/components/IdentitySearchBar.tsx`) の構造を踏襲
  - debounce: 200ms
  - 結果表示: 上限 8件 (現行の `MAX_RESULTS` を維持)
  - 選択時挙動: 該当 Identity の primary home バケットで `IdentityLogSheet` を pre-select 状態で開く

#### F2. 検索ロジック強化
- **正規化**: 入力クエリと Identity の検索対象文字列を以下のルールで揃える
  - 全角 ↔ 半角
  - ひらがな ↔ カタカナ
  - 大文字 ↔ 小文字
  - ローマ字 → ひらがな (一般的な訓令式 / ヘボン式の両対応)
- **同義語辞書**: Identity ごとに `searchTags` (既存スキーマ) で同義語を保持
  - 例: `broccoli` Identity の `searchTags`: `["ブロッコリ", "broccoli", "ぶろっこり"]`
  - 辞書は別ファイル化し、運用で追加できる構造にする
- **部分一致**: 前方一致 + 部分一致で広めに拾う

#### F3. ランキング (頻度学習統合)
- 検索結果の並びは以下を加味:
  1. ユーザー固有の選択頻度 (§6.5.1 自動学習と統合)
  2. グローバルな出現頻度 (Identity マスター側)
  3. 文字列マッチ度 (前方一致を優先)
- 同点時は Identity の primary home バケット順 (タブ並び順と整合)

#### F4. 量入力フロー
- 検索結果をタップ → IdentityLogSheet を開く (現行挙動維持)
- 量入力は Quick Log と同じ `AmountSpec` を使用
- ブランド指定がある Identity は `brandChips` を提示 (PRD §13.1 P2-S4 と統合)

#### F5. 検索結果0件の処理
- 「該当する食事が見つかりませんでした」を中立トーンで表示
- 以下の選択肢を提示:
  - 「カスタム食品を登録」(P2-S7、Next で実装)
  - 「リクエストを送る」(運用フィードバックチャネル)
- `quick_log_unfound_event` を発火 (補助KPI、§6.3)

### 3.2 Should Have (Next 内で実装可能なら)

#### F6. 直近検索履歴
- 検索バー focus 時、直近5件の検索クエリを表示 (再検索ショートカット)

#### F7. 検索カテゴリ絞り込み
- 検索結果上部に「すべて / 9バケット名」のセグメントを表示
- 大量に同名 Identity が出る場合の絞り込み (例: 「茶」検索でドリンクのみに絞る)

### 3.3 Could Have (Next 後半 〜 Later)

#### F8. 音声入力 → 検索バーブリッジ (PRD §13.1 P2-S8)
- iOS / Android 標準 Speech Recognition を検索バーに接続
- 音声 → 検索クエリ変換のみ。AI 解釈は P3-B (Later) で

### 3.4 Won't Have (本仕様の範囲外)

- 写真認識 (P3-A、Later)
- 自然言語解析 ("鶏むね150gとブロッコリー" の分解、P3-B、Later)
- レシピ分解 (P3-C、Later)
- 検索結果からの直接バーコードスキャン起動 (P2-S5 とは別UI)

## 5. 検索ロジック詳細

### 5.1 正規化パイプライン

**実装位置:** `expo/utils/identity-normalize.ts` (新規)。検索クエリ側と Identity 側 (起動時1回) の両方を同じ関数に通す。

#### 5.1.1 正規化ステップ (順序固定)

| 順 | ステップ | 入出力例 | 実装ノート |
|---|---|---|---|
| 1 | `trim()` | `"  チキン  "` → `"チキン"` | 標準 |
| 2 | **NFKC 正規化** (全角英数記号 → 半角、合成文字統一) | `"Ｂｒｏｃｃｏｌｉ１"` → `"Broccoli1"` | `String.prototype.normalize('NFKC')` |
| 3 | `toLowerCase()` (en-US ロケール) | `"Broccoli"` → `"broccoli"` | 標準 |
| 4 | **カタカナ→ひらがな** | `"ブロッコリー"` → `"ぶろっこりー"` | コードポイント計算 (`0x30A1〜0x30F6` → `0x3041〜0x3096`、`-0x60`) |
| 5 | **ローマ字→ひらがな** | `"burokkori"` → `"ぶろっこり"` | 内製辞書 (§5.5)、最長一致でトークナイズ |
| 6 | **長音/促音/拗音の正規化** (検索容易化) | `"ぶろっこりー"` → `"ぶろっこり"` | 長音 `ー` 除去、または直前文字の母音に展開 |

**設計原則:**
- **失敗時は止めない**: 各ステップは例外を投げず、不正入力時は前段の結果をそのまま返す (最低限 trim + lowercase は通過する設計)
- **冪等性**: 正規化済みの文字列に再適用しても結果が変わらない
- **検索クエリ側と Identity 側で同一関数を使う**: 二重実装による乖離を防ぐ

#### 5.1.2 長音 (`ー`) の扱い

- **searchTags 側**: 元表記 + 長音除去版の **両方** をインデックスに登録
  - 例: `"ぶろっこりー"` と `"ぶろっこり"` の両方を保持
- **クエリ側**: 長音を残したまま正規化 (前方一致で両方ヒット可能)

これにより、ユーザーが `"ブロッコリ"` (長音省略) で検索しても `"ブロッコリー"` Identity がヒットする。

#### 5.1.3 想定挙動 (受け入れテストの種)

| ID | クエリ | 期待ヒット |
|---|---|---|
| T1 | `BROCCOLI` | broccoli Identity (label: ブロッコリー) |
| T2 | `ぶろっこりー` | 同上 |
| T3 | `ﾌﾞﾛｯｺﾘｰ` (半角カナ) | 同上 |
| T4 | `burokkori` | 同上 |
| T5 | `salmon` | 鮭 Identity (英語名 searchTags) |
| T6 | `さけ` | 同上 (ひらがな searchTags) |
| T7 | `さば缶` | 鯖缶 Identity (既存 searchTags でヒット) |
| T8 | `だいず` | 大豆 Identity |
| T9 | `coffee` | コーヒー Identity |
| T10 | `ブロッコリ` (長音省略) | ブロッコリー Identity |

#### 5.1.4 パフォーマンス想定

- インデックス生成 (起動時): 3,000 Identity × 平均 5 searchTags = 15,000 文字列の正規化
  - 1文字列あたり 0.05ms 想定 → 起動時総コスト < 750ms (Splash で吸収可能)
- 検索時のクエリ正規化: < 1ms
- 検索結果表示まで: §7.3 の 100ms 以内を維持

### 5.2 マッチング
- 前方一致: スコア ×3
- 部分一致: スコア ×1
- `searchTags` ヒット: スコア ×2
- 表示名ヒット: スコア ×2
- (将来) Levenshtein 距離でのファジー一致: スコア ×0.5 (P2-S2 後半で検討)

### 5.3 ランキング式
```
finalScore = matchScore × (1 + userFreqWeight × log(1 + userSelectCount))
                       × (1 + globalFreqWeight × log(1 + globalSelectCount))
```

- `userFreqWeight`: 0.5 (ユーザー個別の頻度を中程度に重視)
- `globalFreqWeight`: 0.2 (新規ユーザーへのフォールバック)
- 重みは A/B テスト可能な定数として実装

### 5.4 同義語辞書

#### 5.4.1 単一情報源 (Single Source of Truth)

**決定: `Identity.searchTags: string[]` を正とする (新規辞書ファイルは作らない)**

理由:
- 既に Identity 単位で散在しており、Identity 追加と同じ PR で同義語が増える運用が成立しやすい
- `Record<IdentityId, string[]>` 別ファイル化は二重管理になり、Identity 削除時の整合担保が面倒
- `searchTags` の TS 型 (`string[]`) はそのままで運用負荷ゼロ

> v0.1 の §5.4 で提案していた `expo/constants/identity/synonyms.ts` 案は **撤回**。

#### 5.4.2 searchTags に格納する情報の必須/推奨/任意

| 種別 | 必須/推奨 | 例 (Identity: ブロッコリー) | 備考 |
|---|---|---|---|
| **表示名カナ表記** | **必須** | `ブロッコリー` (label と同じ場合は省略可) | 表記揺れの起点 |
| **同名のひらがな表記** | **必須** | `ぶろっこりー` | カタカナ→ひらがな変換が無くてもヒット |
| **同名のローマ字表記 (ヘボン式)** | **必須** | `burokkori`, `burokkoorii` | 日本語入力前提でも素早い検索を可能に |
| **英語名** | **推奨** (主要食材のみ) | `broccoli` | §11 #1 の解決方針: あすけん頻出 × 一般定着 100食材に限定 |
| **省略形/口語/別名** | **推奨** | `ブロッコリ` (長音省略) など | ベンチで頻出する表現を拾う |
| **誤記/表記揺れ** | **任意** | — | 0件クエリログから運用追加 |

#### 5.4.3 0件クエリの運用フィードバックループ

- `quick_log_unfound_event` (§6.3 / PRD §10.2.1) の props に `query` を含める
- Analytics キューから月次で 0件クエリを抽出
- ヒット候補が決まった Identity の `searchTags` に追記する PR を月次で作成
- 該当 Identity が存在しない場合は「Identity DB 拡張 (P2-S3)」候補リストに送る

#### 5.4.4 メンテルール

- Identity 追加時に **最低3つ** の `searchTags` を必須化 (label カナ + ひらがな + ローマ字)
- 例外: label が既にひらがな or 漢字 (例: `ごはん` `卵`) の場合でも、カナ + ひらがな別表記 + ローマ字を必須とする
- 担保方法: PR テンプレ項目に「searchTags ≥ 3 件 (ひらがな + ローマ字を含む)」を追加。中長期では ESLint カスタムルールも検討
- 既存 Identity (115件中 103件が searchTags 未定義 — §6.1 脚注参照) は Phase 2 投入前のデータ整備フェーズで一括補充する

### 5.5 ローマ字→ひらがな辞書範囲

#### 5.5.1 採用方針

**ヘボン式優先 + 訓令式の主要パターンを併存** (両方の入力で同じひらがなに変換する)

理由:
- ヘボン式は実用 (パスポート / 道路標識 / 駅名表示) で広く使われる
- 訓令式は学校教育・公的記録で使われる
- 両方カバーすることで「`shi` でも `si` でも『し』にヒット」を保証する

#### 5.5.2 主要マッピング例

| ローマ字入力 | ヘボン式 | 訓令式 | 採用 |
|---|---|---|---|
| `shi` / `si` | し | し | 両方 → し |
| `chi` / `ti` | ち | ち | 両方 → ち |
| `tsu` / `tu` | つ | つ | 両方 → つ |
| `fu` / `hu` | ふ | ふ | 両方 → ふ |
| `ja` / `zya` | じゃ | じゃ | 両方 → じゃ |
| `kya` | きゃ | きゃ | (同じ) |
| `kk` (促音) | っk | っk | 子音重複から `っ` 生成 |
| `ou` / `oo` (長音) | おう / おお | おう / おお | 両方を保持 (元表記 + 長音除去版) |

#### 5.5.3 辞書サイズと実装

**約 200 エントリ** で網羅可能:
- 清音: 46
- 濁音/半濁音: 25
- 拗音 (kya, sha, cha, ...): 35
- 撥音 (n, nn): 2
- 促音 (kk, tt, ss, ...): 子音重複処理ロジック
- 訓令式エイリアス: ヘボン式と異なるパターンのみ (約 15)
- 長音処理 (ー, ou, oo, uu, ei): 専用ロジック

**辞書ソースの判断:**
- 採用方針: **内製辞書** (200件なら依存追加不要、bundle size への影響ゼロ)
- 既存ライブラリ `wanakana` (npm, ~30KB) は実装ゼロだが依存増 → 当面は内製、複雑化したら再検討

#### 5.5.4 トークナイズ戦略

**最長一致** で先頭から消費する:
1. 入力: `"sashimi"`
2. 先頭から最長 3文字 → 2文字 → 1文字の順でマッチ試行
3. `"sa"` → `さ` (2文字消費), `"shi"` → `し` (3文字消費), `"mi"` → `み` (2文字消費)
4. 結果: `"さしみ"`

トークナイズに失敗した残文字 (例: `"xyz"`) はそのまま結果に残す (検索クエリ側ではゼロマッチを許容する保険として機能)。

[^1]: **既存 searchTags カバレッジ実測値 (2026-05-01 計測, `expo/scripts/analyze-search-tags.ts`)**

## 9. リスクと対応

| リスク | 対応 |
|---|---|
| 検索が主導線化し、9ボタン × 自動学習の効果計測が崩れる | `identity_search_usage_rate` と 9ボタン経由ログ完了率を分離計測。検索利用率が想定 (15〜30%) を大きく超えたら導線設計を見直し |

| ローマ字変換の誤変換 | 訓令式 / ヘボン式の主要パターンのみ対応。エッジケースは部分一致でカバー |
| パフォーマンス劣化 (DB拡張時) | インメモリインデックス + lazy loading。3,000 件超になったら検索エンジン (Fuse.js 等) 導入を検討 |
