## 1. 設計思想 — 5つの根幹原則

| # | 原則 | 何を守るか |
|---|------|-----------|
| ① | **9ボタン × 2タブ構成** | 認知負荷の物理上限を固定。タップ=代表値で即記録、長押し=詳細調整 |
| ② | **PFC収束軸** (食材タブ) | 同一ボタン内の食品はカロリー・PFCが類似。代表値記録の誤差を許容範囲に収める |
| ③ | **ジャンル × スタイル軸** (一皿料理タブ) | 組み合わせでPFCが激変する料理を、ジャンルとスタイルで1タップ精度を担保 |
| ④ | **無カロリー除外** | 水・お茶・ブラックコーヒー・塩・醤油等 PFC無寄与のものは記録対象外。入力ノイズ削減 |
| ⑤ | **専門用語の排除** | 「主食」→「ごはんパン麺」、「低脂P」→「鶏むね魚」など、直感的なラベルのみ採用 |

**ゴール**: ユーザーが直感で選んだ1タップが、栄養学的事実と **90%以上一致** している状態。微細な差 (ホエイかソイか / トーストか生か) は勇気を持って削ぎ落とし、大きな差 (鶏か牛か / 皮ありか無しか / 牛乳割りか水割りか) を確実に捉える構造にする。

## 2. 階層化のルール — 食品データを5層に分解

頭がこんがらがらないための情報優先順位。**Identity → Attribute → Style → Add-on** の責務分離が中核。

### 2.1 第2階層 — Identity (素材) の純化

サブカテゴリには **素材の名称のみ** を置く。「茹で」「焼き」などの調理法は混入させない。

- ✅ `鶏肉` `豚肉` `鮭` `米`
- ❌ `茹で鶏` `焼き鮭` `炒飯`

### 2.2 第3階層 — Attribute (属性) によるベース確定

素材の性質そのもの = **PFCの基礎単価** を決定する要素を、調理法より先に選択させる。

- 鶏肉: `皮なし` / `皮つき`
- 豚肉: `ヒレ` / `ロース` / `バラ`
- 米: `通常` / `玄米` / `おかゆ` / `おむすび`
- 卵: `全卵` / `卵黄` / `卵白`
- 牛乳: `通常` / `低脂肪` / `加糖`

### 2.3 第4階層 — Style (調理法) の動的加算

`焼き` `揚げ` などの調理法は独立した階層で扱い、油の付加量を計算する。**脂質が大きく動くカテゴリでのみ表示** する (米・パン・乳製品では非表示)。

| 表示するカテゴリ | 動的選択肢 |
|---|---|
| 肉・魚・野菜・いも | `生・茹・蒸` / `焼き` (油微増) / `揚げ` (油大幅増) |
| 米・パン・乳製品 | (Style不要 — Attributeでベース確定済み) |

### 2.4 第5階層 — Add-on (トッピング) の厳選

**1単位の追加でPFCが3〜5g以上動くもの** だけを選択肢として提示する。

- ✅ 卵 / 納豆 / マヨネーズ / バター / チーズ / 鮭 / ナッツ
- ❌ 醤油 / 塩 / 七味 / レモン / 大葉 (PFC無寄与または微小)

## 3. 外れ値・多義性の処理ルール

データ整合性を守るための **強制移動ルール**。実装時に必ず事故が起きるエッジケースを先回りで言語化。

### 3.1 PFC崩壊の遮断

ベース素材と調理法の組み合わせが **別カテゴリの料理に変質** する場合、内部的に強制リダイレクトする。

| トリガー | 強制マッピング先 |
|---|---|
| `じゃがいも` (食材) + `揚げ` | `フライドポテト` (一皿料理タブ, 脂質15g) [3: S-029] |
| `鶏むね` (食材) + `揚げ` + 衣 | `唐揚げ` (一皿料理タブ) |
| `豚バラ` (食材) + 大量油 + 麺 | `焼きそば` (一皿料理タブ) |

**目的**: 食材タブの低脂質想定値を保護し、検索結果の整合性を維持する。

### 3.2 マルチエントランス (複数入口の保証)

同一 Identity に対して複数バケットからの導線を許容する。**ユーザーの検索迷子を防ぐ**。

| Identity | 入口A | 入口B | 共通到達点 |
|---|---|---|---|
| プロテインシェイク | `5. 乳・大豆` | `2. 低脂タンパク` | サプリ系タンパク (PR-001) |
| ヨーグルト | `5. 乳・大豆` | `7. 果物` (フルーツ添え) | 乳製品 + Add-on |
| 鶏むね | `2. 低脂タンパク` (皮なしdefault) | `3. 脂タンパク` (皮つきdefault) | 鶏肉 Identity |

### 3.3 Identity重複の運用カバー

重複するIdentityは、**入口バケットによってデフォルト Attribute を切り替える** ことで1タップ精度を担保する。

- `2. 低脂タンパク → 鶏肉` → デフォルト `皮なし`
- `3. 脂タンパク → 鶏肉` → デフォルト `皮つき`
- `2. 低脂タンパク → 魚介` → デフォルト `白身/水煮`
- `3. 脂タンパク → 脂乗り魚` → デフォルト `鯖/サーモン/油漬缶`

### 4.1 表の重要ポイント

1. **Identity重複は運用でカバー** (§3.3 参照): 鶏肉が `2. 低脂タンパク` と `3. 脂タンパク` の両方にあるが、入口で `皮なし` / `皮つき` のデフォルトが切り替わるので、1タップ精度は担保される
2. **麺類は食材タブから排除**: 「うどん」「そば」「パスタ」は一皿料理タブへ完全移籍。食材タブに残るのは自炊用のプレーン素材のみ
3. **Style (調理法) は動的表示**: 米・パン・乳製品など脂質変化が微細なものでは第4階層を表示しない。**肉・魚・野菜・いも** のように `焼き・揚げ` で脂質 (F) が激変するものでのみ出現
4. **Add-on は PFC を持つものだけ**: 醤油・塩・七味などの無カロリー因子は除外。卵・納豆・マヨ等、それ自体に栄養素を持つ食材のみベース素材との相性で提示

## 6. 受け入れ条件

### 6.1 構造として
- 食材タブが上記9バケットで構成され、各バケットの Identity が定義されたとおりに配置されている
- Style (第4階層) は脂質変化が大きいカテゴリでのみ動的に表示される
- Add-on (第5階層) は PFC を持つ食材のみで構成される
- 一皿料理タブが (5.1 の) 9バケットで構成される (Identity 詳細は別タスク)

### 6.2 振る舞いとして
- ユーザーがバケットボタンをタップ → 代表値で即時ログされる
- 長押し → Identity / Attribute / Style / Add-on の階層を順に展開できる
- `じゃがいも + 揚げ` 等の組み合わせは、内部で一皿料理タブ側にリダイレクトされる (§3.1)
- 入口バケットによってデフォルト Attribute が切り替わる (§3.3)

### 6.3 メトリクスとして
- 1タップ完了率 70% 以上 (長押し詳細展開を必要としない記録)
- 1記録あたりの平均タップ数 ≤ 1.5
- ログ後の手動修正率 < 15%
