## 1. 目的

本プロダクトの目的は、ユーザーがアプリ開始直後に「自分の今の体型」と「なりたい体型」を直感的かつ納得感を持って把握し、その内容をもとに、無理のない体重・体脂肪率・摂取カロリー・PFCバランスの初期提案を受けられる状態を作ることです。加えて、初回から食事記録をしやすくするために、オンボーディングで取得した嗜好や食行動の傾向をクイックログUIへ反映し、初期体験の摩擦を下げます。

体型分類は「体脂肪だけ」で見せるのではなく、**脂肪量 × 筋量の二軸** で整理した共通9分類を採用します。これにより、「細いけれど筋肉が少ない」「重いけれど筋肉が多い」など、現実の見た目に近い分類を表現でき、現在体型と目標体型を同じ概念体系で扱えます。分類思想の根拠は、Tanitaの体組成タイプ判定、InBodyの筋肉量・脂肪量の複合評価、日本のBMI/体脂肪率目安に基づいています。

## 2. 対象ユーザー

### 2.1 コアターゲット
ダイエット・ボディメイク・健康管理に関心はあるが、専門的な身体指標には詳しくない一般ユーザー。特に、「痩せたい」だけではなく「引き締めたい」「筋肉をつけたい」「健康的に整えたい」といった、見た目ベースで目標を考える人。

### 2.2 主要ユーザー像
- **A: なんとなく体型を整えたい初心者** — 数値より見た目で判断したい。BMIや体脂肪率だけでは目標が分かりにくい。
- **B: 食事管理を始めたいが入力が面倒な人** — 最初の入力負荷が高いと離脱しやすい。
- **C: 体重だけでなく体型の変化を重視する人** — 同じ体重でも見た目が違うことを理解している。

## 3. プロダクト方針

### 3.1 体型定義の基本方針
現在体型と目標体型は、**同じ9分類マトリクス** で扱う。「現在は9分類、目標は5分類」などの不一致は採用しない。

### 3.2 9分類マトリクス（脂肪 × 筋量）

体型分類は **脂肪3段階 × 筋量3段階** の9セルで定義する。
**命名 (「スリム」「バランス」等) は一切付与しない** — 価値判断や羞恥を生まないために、表記は **軸ラベルの組み合わせのみ** とする。

両軸のラベルは **「少なめ / ふつう / 多め」** で統一する (認知負荷最小化)。

|              | 脂肪 **少なめ**   | 脂肪 **ふつう**   | 脂肪 **多め**     |
|--------------|-------------------|-------------------|-------------------|
| **筋量 少なめ** | fat↓ / musc↓    | fat→ / musc↓     | fat↑ / musc↓     |
| **筋量 ふつう** | fat↓ / musc→    | fat→ / musc→     | fat↑ / musc→     |
| **筋量 多め**   | fat↓ / musc↑    | fat→ / musc↑     | fat↑ / musc↑     |

UI/My Statusでの表記例: 「**脂肪 · ふつう × 筋量 · ふつう**」

#### 9タイルの参考値テーブル

各セルに紐づくBMI中央値とBF%中央値を定義する。オンボーディングUIでは、ユーザーの身長に応じた **参考体重 (= BMI × 身長²)** と **BF%の目安** を動的に表示する。

**男性基準**
|              | 脂肪 少なめ    | 脂肪 ふつう    | 脂肪 多め      |
|--------------|---------------|---------------|---------------|
| 筋量 少なめ   | BMI 18 / BF 12% | BMI 20 / BF 20% | BMI 25 / BF 27% |
| 筋量 ふつう   | BMI 20 / BF 12% | BMI 23 / BF 17% | BMI 26 / BF 25% |
| 筋量 多め     | BMI 22 / BF 10% | BMI 25 / BF 15% | BMI 28 / BF 22% |

**女性基準**
|              | 脂肪 少なめ    | 脂肪 ふつう    | 脂肪 多め      |
|--------------|---------------|---------------|---------------|
| 筋量 少なめ   | BMI 17 / BF 20% | BMI 20 / BF 26% | BMI 24 / BF 33% |
| 筋量 ふつう   | BMI 19 / BF 20% | BMI 22 / BF 25% | BMI 25 / BF 31% |
| 筋量 多め     | BMI 21 / BF 18% | BMI 24 / BF 23% | BMI 27 / BF 29% |

### 3.3 見た目表現の方針
- リアル写真ではなく、中立的な **フラットベクターSVGイラスト**
- 比較可能性 / UI統一性 / 非性的・非評価的 / スケーラビリティ重視
- 基準マスター: **男性M5・女性F5** をシリーズ中心点として固定

## 5. ユーザーフロー

オンボーディングは **1画面1入力原則** で構成する (モダンな健康アプリ標準パターン)。各画面は超軽量化し、**キーボード入力で即次へ / タップで自動次へ** の UX を目指す。

### Phase 0 — App Intro / Paywall (強制課金型)
価値理解 → 無料トライアル理解 → サインアップ/購読
**重要**: paywall は「あとで」スキップ不可。7日間トライアル開始または購読 (or 復元) が必須。

### Phase 1 — 基礎データ (各画面5秒程度)
- **1.1** 性別基準 (2択タップ)
- **1.2** 身長 (数値入力)
- **1.3** 体重 (数値入力)
- **1.4** 年齢 (数値入力)
- **1.5** 運動習慣 (4択タップ、§6.2 参照)

### Phase 2 — 現在の自分
- **2.1** 現在体型の選択 (9分類タイル)
  - 各タイルに **「あなたの身長だと 体重○○kg · BF○○%前後」** の参考値を表示
  - 入力済み体重からBMIを計算し、**近いタイルを控えめにハイライト** (選択は自由)
  - 選択後、体脂肪率が **自動入力される** (表示: "体脂肪率: 17% (目安)")
  - 自動値は編集可能。ユーザーが上書きすると "(目安)" → "(入力済)" に変化
  - **選択結果の派手な演出はしない** (価値判断を避ける)

### Phase 3 — 目標設定
- **3.1** 目標体型の選択 (同じ9分類タイル)
  - 現在位置が視覚的にハイライト
  - 各タイルに目標体重・目標BF%の参考値を表示
  - 現在→目標のベクトルから、方向 (減量/維持/増量) が自動導出される
  - **目標決定時はポジティブな演出を許容** (具体的な演出は別途検討 — TBD)
- **3.2** ペース選択 (ゆるやか / 標準 / しっかり)
- **3.3** 目標数値プレビュー (kcal / PFC / 目標体重) — 手動微調整可

### Phase 4 — 完了
- **4.1** 設定完了サマリー + 次のアクション提示 (最初の食事記録へ誘導) → Home

### 完了後
- My Status Hub / Quick Log開始
- 食事傾向 (時間帯・よく食べるもの・量) は **ログ蓄積から自動学習** されQuick Logに反映される (§6.5)

### 進捗表示
Phase単位で表示する (例: "Phase 1 · 3/5")。全10ステップを単一プログレスバーにするより残タスク感が軽減される。

### フロー全体のステップ数
- **合計 10ステップ** (Phase 0 Paywall 以降)
  - Phase 1: 5ステップ (基礎データ)
  - Phase 2: 1ステップ (現在体型)
  - Phase 3: 3ステップ (目標体型 / ペース / 数値プレビュー)
  - Phase 4: 1ステップ (完了)

## 6. 機能要件

### 6.1 App Intro / 課金導線

**料金モデル**: **強制課金型 + 7日間無料トライアル**
- 月額 ¥480 / 年額 ¥4,800 (年間 17% 割引)
- 7日間トライアル後、自動課金 (Auto-renewing subscription)
- トライアル/購読いずれかの状態でないとアプリ機能を使えない (paywall は「あとで」スキップ不可)

**要件**
- 3〜4枚の導入スライド
- 有料モデル、無料トライアル期間 (7日)、解約可否を明示
- サインアップ導線と復元導線
- 利用規約・プライバシーポリシーへのリンク
- トライアル終了 2日前にローカル通知で告知 (ユーザー体験保護)
- プロフィール画面 (`/status`) にトライアル残り日数を常時表示
- 残り 2日以下になったらホーム画面アバターアイコンにバッジ表示

**受け入れ条件**
- 初回起動時に App Intro → onboarding → paywall の順で表示される
- 未課金/未トライアルでホーム到達不可 (paywall を必ず通る)
- 「購読する」「復元する」「規約確認」が機能する
- 購入完了時にローカル通知の許諾フローが起動する
- トライアル中は `/status` 画面で残り日数が常時確認できる

### 6.2 基礎データ取得 & 現在体型の選択

#### 6.2.1 入力項目
性別基準 / 身長 / 体重 / 年齢 / **運動習慣** / 現在体型 (9分類) / 体脂肪率 (体型選択から自動入力、上書き可)

#### 6.2.2 現在体型の選択UI (自動推定ではなくユーザー選択)

従来の「4項目から自動推定」方式ではなく、**ユーザーが見た目で9分類から選ぶ** 方式を採用する。理由:
- 体脂肪率を知らないユーザーの離脱を防ぐ
- PRD想定ユーザーA (「数値より見た目で判断」) にフィット
- 9分類マトリクスを中核UX (PRD §3.2) として主役化できる

**UI仕様**:
- 9タイルを3×3のマトリクス表示 (イラスト+軸ラベル+参考値)
- 各タイルに、ユーザー入力値から動的算出した **参考体重 (身長から)** と **BF%目安** を表示
  - 参考体重 = `セルのBMI × (身長/100)²`
  - BF% = §3.2 テーブル値
- ユーザーの現在BMIから**近似タイルを控えめにハイライト** (選択は自由)
  - 距離 = `|userBMI - tileBMI| / 3`
- タイル選択後、**BF%が自動入力される** (タイル中央値)
- BF%は下部のフィールドで編集可: "体脂肪率 17% (目安) ▸" → タップで入力モードに
- ユーザーが手入力すると "(目安)" → "(入力済)" 表示に変化
- **派手な結果演出はしない** (静的表示のみ)

#### 6.2.3 運動習慣の取得 (1問)
PRD §6.4 のマクロ目標算出に活用する。体型選択そのものには使わない。

| Lv | 選択肢 | 想定する人 | 活動係数 | タンパク質目安 (維持時) |
|----|--------|-----------|---------|---------------------|
| 1 | ほとんど運動しない | デスクワーク中心・座りがち | 1.20 | 1.2 g/kg |
| 2 | 軽めに動く | 週1–2回の散歩・軽い運動、立ち仕事 | 1.38 | 1.4 g/kg |
| 3 | よく運動する | 週3–4回の運動 or 筋トレあり | 1.55 | 1.6 g/kg |
| 4 | しっかり鍛えている | 週5回以上・筋トレ中心・スポーツ | 1.73 | 1.8 g/kg |

#### 6.2.4 例外処理
- 身長・体重・年齢・性別・運動習慣が未入力の場合、Phase 2 に進めない
- 体型未選択時は Phase 3 に進めない
- BF%はタイル選択から自動入力されるため、明示的未入力は発生しない
- 運動習慣未選択時は進行不可 (または Lv1 フォールバック — フロー決定時に確定)

#### 6.2.5 受け入れ条件
- 基礎データ 5項目入力後、Phase 2 に進める
- 9タイルには参考値が表示され、近似タイルがヒント表示される
- タイル選択で体型が一意に確定し、BF%が自動入力される
- BF%は手動上書き可能
- 数値編集時に参考値・ヒント・提案値が再計算される
- 運動習慣は1問4選択肢で取得される

### 6.3 目標体型の決定 (目的 + プラン)

**v1.3 変更**: 目標体型を 9-マトリクスから手動選択する UI は廃止し、**目的 (減量/維持/増量) 1問** + **ペース選択 (3択)** から自動導出する方式に統一した。ユーザーの認知負荷を下げ、目的 ↔ 目標セルの整合性を構造的に保証する。

**UI仕様** (オンボ / goal-edit 共通):
- **目的 (direction)** を 3択で選択: `減らす / 維持 / 増やす`
- 選択された目的から目標セルを `deriveTargetCellFromDirection(current, direction)` で自動導出
  - 減量: 脂肪軸を 1段階下げる (既に最小なら現セル維持)
  - 維持: 現セル
  - 増量: 筋量軸を 1段階上げる (既に最大なら脂肪軸を 1段階上げる = リーンバルク)
- 目標体重/BF% は §6.4 の `computePlanOutcome` により **ユーザーの現在値 × ペース** から計算される
  - **(v1.7)** ただしこの試算値は**コミット時に一度だけ固定アンカーとして確定**し、以後の体重変化や目的/ペースを変えない再保存では動かさない (ドリフト防止規則 → §6.4.4)
- オンボでは「**3ヶ月後、どう変わりたい？**」として 3プランカード (ゆるやか/標準/しっかり) を並列比較表示する
  - 各カードに 3ヶ月累計 kg差 / 月次目安 / BF%変化 / 目標体型シルエット / "体型が変わるペース" or "変化は小さめ" ヒント
  - 標準プランには **「おすすめ」バッジ**

**ポジティブな演出について**:
目標確定時は、ユーザーのコミットを後押しするために**ポジティブな演出を許容** する (例: アクセントカラーのアニメーション等)。具体的な演出設計は別論点として別途検討 (TBD)。

**受け入れ条件**:
- 目的を選ぶだけで、目標セル・目標体重・目標BF%・kcal/PFC が一意に決まる
- プランを切り替えると上記が即座に再計算される
- 目的「維持」選択時はペース選択をスキップ (ペース無関係)
- **(v1.7)** 目的・ペースを変えずに goal-edit を再保存しても目標体重は変化しない (固定アンカー, §6.4.4)

### 6.4 目標数値提案

#### 6.4.1 算出ロジック

**RMR (基礎代謝)**
- 体脂肪率 **未入力** の場合: Mifflin-St Jeor 式
  - 男性基準: `9.99×体重 + 6.25×身長 - 4.92×年齢 + 5`
  - 女性基準: `9.99×体重 + 6.25×身長 - 4.92×年齢 - 161`
- 体脂肪率 **入力あり** の場合: Katch-McArdle 式 (精度優先)
  - `370 + 21.6 × LBM` (LBM = 体重 × (1 - 体脂肪率/100))

**TDEE (消費カロリー)**
- `TDEE = RMR × 運動習慣の活動係数` (§6.2 Lv1〜4)

**目標カロリー**
- 基本: **体重%ベース** で deficit / surplus を決める
  - 減量: `TDEE - (体重 × 7.7 × 減量ペース係数)` ≒ 体重の **0.5–1.0%/週減** に相当
  - 維持: `TDEE`
  - 増量: `TDEE + (体重 × 3.85 × 増量ペース係数)` ≒ 体重の **0.25–0.5%/週増** に相当
- 最低保証: 男性1500kcal / 女性1200kcal を下限としてクリップ
- ペース選択「ゆるやか / 標準 / しっかり」で係数を 0.5x / 1.0x / 1.5x 調整

**タンパク質 (g)**
- 体重kgベース:
  - 維持: 運動習慣のP目安 (1.2〜1.8 g/kg)
  - 減量: 維持値 + **0.3 g/kg** (筋肉温存)
  - 増量: 維持値 + **0.2 g/kg**
- 計算結果の kcal% は 13–35% の範囲で自動クリップ

**脂肪 (g)**
- 下限ガード: **体重 × 0.6 g/kg** または 総kcal × 20% のうち大きい方
- 減量中の上限: 総kcal × 30%

**炭水化物 (g)**
- 残りカロリーから算出: `(目標kcal - P×4 - F×9) ÷ 4`

**目標体重 / 目標BF% (v1.3)**
- 優先順位1: **プラン試算** (`computePlanOutcome`) — 現在体重/BF% × ペース係数で 3ヶ月後を予測
  - 減量: 週次 -0.25% / -0.5% / -0.75% × 13週 → 体重差 → 最終体重
  - BF%: 減量時は 75% が脂肪量減と仮定して再計算
  - 増量: 週次 +0.125% / +0.25% / +0.375%、30% を脂肪増と仮定
- 優先順位2: **セル参考値** (`getCellRef` の BMI 中央値 × 身長²) — ペース未選択時のフォールバック
- **維持時 (v1.7 改訂)**: §6.4.4 の切替ルールに従う。減量ゴール未到達なら「減量継続 / 現体重で維持」を確認。到達済み・初回維持なら現在体重をアンカー
- **(v1.7) 上記試算は「予測」。コミットで確定した値は固定アンカーとして凍結**し、体重変化・単純な再保存では動かさない → §6.4.4

#### 6.4.3 動的TDEE — 活動量による日次目標補正 (差分方式, v1.8)

> Health Connect / Apple Health から取得した当日の活動量を、日次のkcal目標へ **差分方式** で反映する。
> 設計の前提は ROADMAP B レーン参照。**全額加算はしない** — 二重計上を避けるため、係数が想定する基準活動量を超えた分のみ上乗せする。

**前提: 活動係数に既に含まれる活動量**
- `TDEE = RMR × 活動係数 (§6.2 Lv1〜4)`。この活動係数は「日常移動を含む1日全体」を表す標準PAL値。
- したがって**活動係数が想定する1日のアクティブエネルギー (基準活動量) は次で求まる**:
  - `baselineActiveKcal = RMR × (活動係数 − 1)`
  - 例: RMR 1,500 / Lv1 (1.2) → 基準 300 kcal、Lv3 (1.55) → 基準 825 kcal
- この基準分は **既に targetCalories に織り込まれている**。歩数/activeKcal を全額足すと二重計上になる。

**差分の算出 (基準超過分のみ加算)**
- 当日の実測アクティブエネルギーを `measuredActiveKcal` とする (Health の Active Energy。歩数しか取れない場合は `歩数 × 体重kg × 0.0005 kcal/歩` で近似)。
- `activityExcess = max(0, measuredActiveKcal − baselineActiveKcal)`
- 普段の日: measured ≈ baseline → excess ≈ 0 → 加算なし → 目標が安定。
- 登山・旅行等: measured が基準を大きく超える → excess を加算。

**運動ログとの二重計上調停 (必須)**
- `measuredActiveKcal` には手動記録した運動 (ExerciseLog) の消費も含まれる。両方を足すと運動分が二重計上される。
- 調停ルール: **「活動由来の加算」と「運動ログ由来の加算」の大きい方を採る**。
  - `exerciseAddition = Σ ExerciseLog.netKcal (当日)` (現行 `adjustedTargetKcal` の挙動)
  - `activityAddition = min(activityExcess, DAILY_CAP)`
  - `finalAddition = max(exerciseAddition, activityAddition)`
- 根拠: activeKcal は運動も日常も包含するため、Health が完全に取れていれば activityAddition ≥ exerciseAddition となり運動分を含む。Health 未連携・未取得なら activityAddition = 0 で **現行挙動 (運動ログのみ) に自動フォールバック** = 後方互換。

**日次キャップ (異常値吸収)**
- `DAILY_CAP` = 800 kcal (GPSノイズによる5万歩等の異常値を吸収。実用上は登山20,000歩でも基準超過 ~600-800kcal に収まる想定)。

**最終目標**
- `adjustedTargetKcal = targetCalories + finalAddition`
- リング分母・「残り kcal」表示に反映。UX 文言は「今日 +○○ kcal の余裕」(目標の置き換えではなく上乗せとして見せる)。

**歩数の別用途 (係数見直しサジェスト)**
- 日次加算とは独立に、直近の平均 `measuredActiveKcal` が選択中の活動レベルの基準から継続的に乖離している場合、活動係数の見直しを中立トーンで提案する (PRD §9.1 準拠)。これにより歩数誤差を「日次の不安定な加算」ではなく「恒常的な係数精度の向上」に効かせる。

**境界 (In Scope / Out)**
- In: 当日活動の差分加算、運動ログ調停、日次キャップ、係数見直しサジェスト。
- Out (§4.2 維持): 心拍ベースの精密TDEE、医療的な代謝判定。

#### 6.4.4 目標アンカーとドリフト防止 (v1.7)

> **背景**: v1.3〜v1.6 では `targetWeightKg` / `targetBodyFatPct` を「現在体重 × ペースの 3ヶ月試算 (`computePlanOutcome`)」で**都度算出し、目標を再保存するたびに現在体重を起点に引き直していた**。このため減量中に目標を編集・再保存すると最終ゴールが手前にずれ (例: 80→73kg を目指していたのが、77kg 時点の再保存で約 74kg へ後退)、特に「減量→維持」切替時は維持目標が**現在体重に無言で固定**され、ユーザーが当初描いたゴールが失われていた。競合 (MacroFactor / MyFitnessPal / あすけん) はいずれも**目標体重をユーザー確定の固定アンカー**として保持し、現在体重から引き直さない。本節でこの方針に統一する。

**概念の分離 (予測 ≠ 目標)**
- **予測 (forecast)**: 「このペースなら 3ヶ月で約○kg」のライブ試算 (`computePlanOutcome`)。現在体重に追従して**動いてよい**。あくまで情報提示。
- **目標 (committed goal / anchor)**: ユーザーがコミットした最終ゴール。**確定時に一度だけ固定し、以後の体重変化や目標の単純な再保存では動かさない**。
- 現状この 2 概念を `targetWeightKg` 単一フィールドが兼務しているのがドリフトの根因。アンカーとして再定義する。

**スナップショットは持たない (簡素化, v1.7)**
- 当初検討した `initialWeightKg` / `initialBodyFatPct` (旅の出発点) は**導入しない**。進捗を「出発点 → 現在 → 目標」の 3 点バーで見せる方針を撤回し、**体重/BF% の推移チャート + 目標ライン**に一本化したため (§6.4.2 / §6.6 進捗表示)。出発点はチャート上の過去データとして自然に写る。
- ドリフト是正の本体は **`targetWeightKg` / `targetBodyFatPct` を凍結アンカー化する一点**に集約する。新フィールド追加は原則不要。
- `goalStartedAt: string` (目標設定日) のみ、必要なら任意で保持してよい (UI 必須ではない)。

**確定 (commit) と更新規則**
- `targetWeightKg` / `targetBodyFatPct` を更新するのは、**ユーザーが目的 (direction) もしくはペース (pace) を実際に変更して保存した時のみ** (= 明示的な再コミット)。
- 目的・ペースを変えずに goal-edit を保存しても、**目標アンカーは据え置く** (ドリフト禁止 / no-op)。変更がなければ保存ボタンを非活性にしてよい。
- 体重・体脂肪の記録 (`addWeightEntry` / `BodyFatEntry`) は `currentWeightKg` / `currentBodyFatPct` のみ更新し、**目標体重には一切触れない** (現行どおり)。
- `targetCalories` / PFC は最新の代謝に追従させる意義があるため現在体重ベースの再計算を許容するが、**目標"体重"アンカーとは別管理**として混同しない。

**減量・増量の目標体重**
- コミット時に `computePlanOutcome(現在体重, ペース)` で最終体重を一度算出し、**アンカーとして凍結**。
- goal-edit 再表示時は凍結済みアンカーを表示。ペース変更時のみ再算出 (= 新アンカーで再コミット)。

**目標体重の手動指定 (任意 / 逃げ道, v1.7)**
- 既定フローは **目的＋ペースからの自動導出**のまま (初心者の認知負荷・ウェルネストーン・9マトリクス整合を守る)。手入力は**主役にしない**。
- 具体的な数値目標を持つユーザー向けに、目標体重を直接指定できる。入力した値が `targetWeightKg` アンカーになる。

  **動線 (確定仕様)**:
  1. **入り口** = プレビューカードの**目標体重ヒーロー数値に編集アフォーダンス (✎)**。タップ編集に入った瞬間が手入力モードへの切替 (「モード」という語はUIに出さない)。数字キーパッド入力 + ±0.5kg ステッパー。
  2. 手入力モードでは **目的セグメントを隠し**、入力体重 vs 現在体重から目的 (減/維/増) を**自動導出**する (矛盾の余地を消す)。
  3. **ペースは残す** (速度＝kcal)。手入力した行き先に対し「標準ペースで約○ヶ月の目安」と結果 (ETA) を情報提示する。期限主導モード (date→pace 逆算) ではない。
  4. カード上に「自分で設定 ｜ **おまかせに戻す**」を表示。戻すと手入力を破棄し目的セグメント再表示・target 再計算で既定へ復帰。
  5. **維持の特例**: 入力値＝現在体重で自動導出が「維持」になる場合はペースも隠す。数値自体が明示的な意思表示のため、§6.4.4 維持切替の「減量継続/現体重維持」確認ダイアログはスキップする。
- **設計境界 (安全)**: 手入力が変えるのは**行き先 (アンカー) のみ**。`targetCalories` / PFC は従来どおり「現在体重 × ペース」から算出し、**目標体重では変えない** → 目標を低く入れても危険な低カロリーにならない (目標＝行き先, ペース＝速度を分離)。
- **ガードレール (エビデンスベース)**: 目標体重の BMI で3段階に判定する。「現在の状態」ではなく「目指す目標」を弾く設計 (摂食障害域を正当化しないため)。
  | 区分 | BMI 範囲 | 挙動 | 根拠 |
  |---|---|---|---|
  | 普通体重 | **18.5〜25** | 通常 | 日本肥満学会 (普通体重)・WHO |
  | ソフト警告 | **<18.5** または **>25** | 事実のみ表示・**保存可** (amber) | WHO 低体重 / JASSO 肥満境界 (日本主対象のため上限は欧米の27/30でなく25) |
  | ハード制限 | **<17.5** | **保存不可** | 摂食障害の臨床紹介基準 (MARSIPAN / RCPsych)。目標を臨床紹介域に置かせない |
  - 上限のハード制限は健康上**設けない** (高い目標体重は飢餓を促さない)。入力欄の既存レンジ (30〜200kg) でサニティのみ担保。
  - **二次防御**: 既存のカロリー下限 (男1500 / 女1200, §6.4.1) を維持し、目標体重が低めでも摂取kcalが危険値にならないよう二重化する。
  - heightCm 未設定時は BMI 算出不可のため判定をスキップ (警告・ブロックなし)。
  - 体型セル / 目標BF% は手入力体重と矛盾しないよう整合させる (セルは見た目補助、アンカーは数値優先)。
  - 出典: WHO BMI 分類 (severe thinness <16 / 紹介・緊急域)、日本肥満学会 肥満度分類 (普通体重 18.5〜25)、MARSIPAN (紹介 <17.5)。

  **文言 (確定 / 中立・非評価トーン)**:
  | 要素 | 文言 |
  |---|---|
  | 既定タグ / 手入力タグ | `おまかせ` / `自分で設定` |
  | 復帰リンク | `おまかせに戻す` |
  | 差分 (減 / 増 / 同値) | `現在 77.0 → 目標 73.0（−4.0 kg）` / `…（＋3.0 kg）` / `現在の体重を維持します` |
  | ETA (ペース名入り) | `標準ペースで約 2.5 ヶ月の見込み` (`ゆるやか`/`しっかり` も同型・ヶ月は0.5刻み丸め・1ヶ月未満=`約 3 週間`・12ヶ月超=`1 年以上`・維持時は非表示) |
  | 健康レンジ警告 (低 <18.5 / 高 >25) | `標準的な体重の目安より低めです（BMI 17.9）` / `標準的な体重の目安より高めです（BMI 26.4）` (事実のみ・amber・保存可) |
  | ハード制限 (<17.5) | `健康的な目安を大きく下回るため、この値では設定できません` (保存不可) |
  | 初回ヒント (任意) | `数値はタップで変更できます` (初回 goal-edit 表示時に一度のみ) |
  | ステッパー a11y | `0.5kg 減らす` / `0.5kg 増やす` (ボタン表示は `−` / `＋`) |
- 手入力で確定した値も**凍結アンカー**として扱い、以後の体重変化で動かさない (上記 commit 規則と同じ)。
- **スコープ外 (バックログ)**: 目標BF% / カロリーの手入力、「○月○日までに」期限逆算モードは本 v1.7 には含めない (§13 / Later)。

**維持 (maintain) への切替**
- 維持切替時、基準体重を**現在体重へ無言固定しない**。直前に減量/増量ゴール (`targetWeightKg` アンカー) が存在し、**現在体重がまだそのゴールに到達していない**場合、**一度だけ確認 UI** を出す:
  - ① 「減量を続ける (目標 例 73kg のまま)」 → **目的は `lose` のまま据え置き**。維持には切り替えない (デフォルト)
  - ② 「今の体重 (例 77kg) を維持に切り替える」 → 目的 = `maintain`、アンカー = 現在体重
- 「当初ゴールまで整えてから維持」という中間状態は**作らない**。ゴールを目指す過程はあくまで `lose` (減量) として扱う。
- 現在体重が既にゴール到達済み、または直前ゴールが無い (初回から維持) 場合は確認なしで現在体重をアンカーとする。

**受け入れ条件**
- 目的・ペースを変えずに goal-edit を保存しても、目標体重が変化しない。
- 体重を記録しても保存済みの目標体重が変化しない。
- 減量ゴール未到達で維持を選ぶと「減量継続 / 現体重で維持」の確認が出て、当初ゴールを無言で失わない。
- 進捗は推移チャート + 目標ラインで表現し、目標は時間が経っても揺れない。
- 目標体重を数値で手入力でき、その値がアンカーになる。健康レンジ外は警告が出る。手入力しても目標カロリーは危険値にならない (行き先のみ変更)。

### 6.5 Quick Log

> **Quick Log の静的構造 (9バケット×2タブ / 5階層 Identity-Attribute-Style-Add-on / 外れ値処理ルール) は [QUICK_LOG_DESIGN.md](./QUICK_LOG_DESIGN.md) で定義する。**
> 本セクションは、その静的構造の上に乗る **動的最適化 (自動学習)** の仕様を扱う。

#### 6.5.0 入力経路のスコープとベンチマーク方針 (v1.6)

##### MVPスコープ
MVP では Quick Log の **9 ボタン × Identity ツリー (長押しで階層展開) × 自動学習 (§6.5.1)** のみで食事入力体験を構築する。Identity 全件検索バーや代替入力経路 (バーコード / 写真AI / 音声 / 自然言語等) は [ROADMAP.md](./ROADMAP.md) で段階導入する。9 ボタンで見つからない Identity に当たった場合のフォールバックは「**長押し → 階層展開**」で吸収する。

##### ベンチマーク方針: あすけん主軸 + 解釈A (v1.6)
検索/入力体験の充実度は、**国内シェア最大の競合「あすけん」を主軸ベンチマーク**に据える。ただし「DB件数で勝負しない」を原則とし、以下の **解釈A** を方針として確定する。

- **コア導線は変えない**: 9ボタン × Identity ツリー × 自動学習が主導線。検索は「ボタン+長押しで届かなかった時の最後の砦」として位置付ける。コア体験の軽さを毀損しない設計を最優先する。
- **検索DBの規模感**: 1,000〜3,000 Identity + ブランドチップ数百〜千件で十分とし、あすけんの 10万件規模は追わない。
- **判定指標**: ユーザーが「あすけんなら載っているのに、ここにはない」を感じる頻度を `quick_log_unfound_event` (§10.2.1) で計測し、しきい値超過時のみ Identity DB を拡張する。
- **拡張優先順**: 「あすけんで頻出 × 自社未収録」リストを起点に、日常的な日本食 / コンビニメニュー / 外食チェーンメニュー / 加工食品の順で投下する。

##### 9ボタンと検索の役割分担
| 経路 | 役割 | カバー想定 |
|------|------|-----------|
| 9ボタン × Identity ツリー | 主導線 | 日常的な食事の 70〜80% を1〜2タップで完了 |
| 長押し → 階層展開 | 第1フォールバック | ツリー内で見つかるが頻度学習に乗っていない Identity |
| 検索バー (Next で再有効化) | 第2フォールバック (最後の砦) | ツリーに含まれないブランド・商品名・特殊な料理 |
| バーコード (Next 後半) | 補助 | パッケージ食品・飲料 |

##### 計測と判断
9 ボタン × 自動学習だけでは届かないケースの規模は §10.2.1 の補助指標 (`quick_log_unfound_event` / `manual_correction_rate` / `long_press_drilldown_rate`) で継続計測する。検索バー再有効化後は `identity_search_usage_rate` も追加計測し、検索→ログ完了率を四半期単位でレビューする。


#### 6.5.1 食事傾向の自動学習

**方針の転換 (v1.2)**:
従来のオンボーディング内での食事パーソナライズ取得は廃止し、**ユーザーの食事ログから自動的に傾向を導出する** 方式に変更する。

**理由**:
- 食事の傾向はユーザーに質問しなくても記録データから推定できる (ログタイムスタンプ・選択食品・量)
- オンボでの質問は「仮説」にすぎず、ログ2〜3日分あれば実データが優先されるべき
- オンボ長さを圧縮することで完了率を改善 (13 → 10ステップ)
- PRD §3.1 方針 (「ユーザーの記録行動で成立する」) と整合

**自動学習の入力 (ログデータ)**:
- 食事ログのタイムスタンプ (朝/昼/夕/間食の分布)
- 選択された Dish / Food の頻度
- 各ログの量

**自動学習の出力 (Quick Log最適化)**:
- Quick Log 初期タブ順 — 各時間帯で頻出カテゴリを優先
- 食材/料理候補の優先表示順 — 頻度順で並び替え
- 量の初期値 — ログ平均量を採用
- おすすめチップ/ショートカット候補 — 実績ベース

**初日体験 (ログ0件時) のフォールバック**:
- デフォルト順 (万人向け一般的なタブ順)
- 量は中央値 (ふつう相当)
- 初回ログ完了後、翌日以降から徐々に学習結果が反映される

**学習のリセット/手動調整**:
- My Status から「学習をリセット」可能 (必要に応じてP2で実装)
- ユーザーが並び順を手動変更した場合は手動優先

**受け入れ条件**:
- ログ蓄積後、Quick Log のタブ順・候補順・初期量がユーザーのログに合わせて変化する
- ログ0件の初日はデフォルトで動作する
- オンボーディングで食事に関する質問は一切行わない

### 6.7 Review (振り返り)

#### 6.7.1 設計原則
記録継続のモチベーションを保つため、過去の自分の積み上げを **軽量に振り返れる** UIを提供する。長期傾向分析や比較レポートは引き続き Out of Scope (§4.2) とし、本セクションでは MVP 範囲の最小限のレビュー体験のみを定義する。

#### 6.7.2 Home横スワイプ (過去日表示・遡及記録)
- Home画面を横にスワイプすると、当日 ← → 過去日に切替わる
- スワイプ可能範囲は **オンボーディング完了日 〜 今日**
- ヘッダー中央に現在表示中の日付ラベル ("今日" / "昨日" / "4/22 (火)") を表示
- 過去日ページの動作:
  - StatusCard (その日の集計) を表示
  - **直近7日以内**: Quick Log を表示。タップで **その日付にログを追加** (timestampは過去日付の現在時刻)。インラインで編集可能なログリスト (タップで編集、削除ボタン)。「○月○日に記録します」のヒントバナーを表示
  - **7日より前**: 読み取り専用リスト。新規追加・編集不可。「7日以上前の日付には記録を追加できません」と表示
  - Today's log bottom sheet は当日表示時のみ (過去日はインライン)
- 当日ページは従来通り (StatusCard + Quick Log + Today's bottom sheet)
- 過去日への記録の上限日数は定数 `MAX_PAST_LOGGING_DAYS = 7` で管理

#### 6.7.3 実績画面 (/stats)
Home ヘッダー右上の実績アイコン (グラフ) からアクセス。週/月のセグメントコントロールで切替。

**週ビュー**
- 週ヘッダー: ← `M/D – D` →。オンボ完了週より前は遡れない、来週は進めない
- **日別 kcal 棒グラフ** (7本)。目標カロリーを点線で重ねる
- **週平均カード**: 平均 kcal/日、平均 PFC、記録した日数 (例 5/7日)
- **日別リスト**: 各日 "M/D (曜) · ◯◯◯ kcal · P◯ F◯ C◯"。タップでHome のその日へ移動

**月ビュー**
- 月ヘッダー: ← `YYYY年 MM月` →
- **カレンダーグリッド** (7列 × 6週、Sunday-start)
  - 各セル: 日付番号 + その日の kcal に比例する円
  - 円の色: 目標±10%内=サージュ系、未達=薄色、超過=暖色
  - 記録のない日は円なし
  - 翌月にまたがるセルは薄表示、未来日はタップ不可
- **月平均カード**: 平均 kcal/日、平均 PFC、記録した日数

#### 6.7.4 導線変更
Home ヘッダー右上の **設定アイコンを実績アイコンに置き換え** る。設定画面へは My Status Hub から遷移する (Progressive Disclosure に整合)。

#### 6.7.5 受け入れ条件
- Home画面で横スワイプして過去日を閲覧できる
- 直近7日以内の過去日では Quick Log で **遡及記録** ができ、その日のキーで保存される
- 8日以上前の過去日では Quick Log が表示されず読み取り専用となる
- オンボ完了日より過去にはスワイプできない
- Home 右上のアイコンから /stats にアクセスできる
- 週ビューでは棒グラフ・週平均・日別リストが正しく表示される
- 月ビューではカレンダーに kcal 量に比例する円が表示される
- 月ビューで日付タップ → Home の当該日付ページに戻れる
- My Status から設定画面にアクセスできる

## 9. 非機能要件

### 9.1 UX
- 片手操作前提、**1画面1入力原則** (モダン健康アプリ標準パターン)
- キーボード入力は returnKey → 自動次へ、タップ選択は選択即次へ
- 数値・説明文・ビジュアルで補完
- **中立トーン**: 価値判断や羞恥を生む命名・表現を避ける
  - 9分類に固有名称 (「スリム」「隠れ脂肪」等) は **付与しない**
  - 軸ラベル (脂肪×筋量 · 少なめ/ふつう/多め) のみで表現
- 各画面のサブタイトルに **「→ 〇〇の計算に使います」** 形式で用途を明示 (データ要求の根拠を見せる)
- **結果フィードバック**:
  - 現在体型選択時: 派手な演出なし (静的表示)
  - 目標体型・目標確定時: ポジティブな演出を許容 (具体は TBD)
- 進捗表示は **Phase単位** (例: "Phase 1 · 3/5") で残タスク感を軽減
- **Progressive Disclosure** (§6.6): 情報は必要な粒度でだけ露出する。ユーザー編集不要なデータ (食事傾向の自動学習結果など) は UI に出さない
- 編集UIは基本 **ボトムシート** で完結。重い編集 (プロフィール等) のみ別画面プッシュ

### 9.2 Performance
- 現在体型再計算はほぼ即時
- SVG読み込みで重くならない

### 9.3 Reliability
- 途中離脱時も再開可能
- 保存失敗時は再試行 or 一時保持

### 9.4 Accessibility
- 色依存しない選択状態
- 文字サイズ可変対応
- 音声読み上げラベル
- テキスト説明併記

### 9.5 Privacy / Compliance
- 取得データは最小限
- 利用目的明示
- 編集・削除可能
- 健康関連データとしての慎重な取り扱い

## 11. 受け入れ条件

### 11.1 オンボーディング
- App IntroからOnboarding完了まで通過できる
- 途中離脱後も再開できる
- 必須入力未完了時は案内が出る

### 11.2 体型ロジック
- 基礎データ (身長/体重/年齢/性別/運動習慣) 入力後、現在体型タイルが選択可能になる
- 現在体型と目標体型が同じ9分類体系で表示される
- 9タイルには参考体重・BF%が動的に表示される
- 現在BMIから近似タイルがヒント表示される
- タイル選択でBF%が自動入力され、手動上書きできる
- 数値修正時に参考値・ヒント・提案値が再計算される
- 9分類には **命名を付与しない** (軸ラベル表記のみ)

### 11.3 目標提案
- 目標体型選択後に目標数値が提案される
- 編集・保存できる
- 極端設定はそのまま確定されない

### 11.4 食事傾向の自動学習
- オンボで食事に関する質問は行わない
- ログ蓄積後、Quick Logのタブ順/候補順/初期量に反映される
- タブ順・候補順・量初期値のいずれか複数が変化する
- 実ログ反映の拡張性を持つ

### 11.5 ビジュアル
- 18体のイラストに統一感
- M5/F5が中央基準として成立
- 特定体型への演出偏りなし

### 11.6 設定管理
- My Statusで確認・編集できる
- 編集結果が関連ロジックに反映される
- 規約/プライバシー/課金復元導線が機能する

## 12. リスクと対応方針

| リスク | 対応 |
|--------|------|
| 体型分類が"診断"に見える | 医療診断ではなく一般的な目安である旨を明示。サブタイトルに「目安づくりに使います」と記載 |
| 体型表現が価値判断・羞恥を生む | **固有名称を付与せず、軸ラベル表記のみ** (§3.2)。イラスト非性的化、比較可能なUIトーン。現在体型選択時は派手な演出なし |
| 体脂肪率未入力ユーザーが多い | **体型タイル選択で自動入力** (§6.2.2)。正確値を知るユーザーは上書き可能 |
| 提案ロジックの説得力不足 | 体型→目標→数値の関係を短文で可視化。各画面サブタイトルで用途を明示 (§9.1) |
| TDEE過小評価による目標精度低下 | **運動習慣を1問取得** (§6.2.3) し、4段階の活動係数を適用 |

## 14. デザイン/開発への引き継ぎメモ

**デザイン**
同一シリーズ比較性が最重要。個別イラストの完成度より、18体を並べたときの一貫性が優先。M5/F5はシリーズの中点であり、そこから端点と中間を差分展開する前提で進める。

**開発**
体型ロジック・目標提案ロジック・Quick Log初期表示ロジックを疎結合に保ち、ロジック差し替えやABテストがしやすい構造で実装する。
