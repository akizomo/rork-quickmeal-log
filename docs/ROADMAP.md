> このドキュメントはMVP リリース後 〜 12ヶ月の機能拡張計画を Now / Next / Later × 5スイムレーン で整理したものです。
> プロダクト方針・機能要件は [PRD.md](./PRD.md) を北極星として参照してください。

> - バージョン: v1.0
> - 最終更新: 2026-04-29
> - 関連: [PRD.md](./PRD.md) §13.1 (食事追加方法ロードマップ) / [SEARCH_SPEC.md](./SEARCH_SPEC.md) (検索機能仕様)

## 1. 全体方針

### 1.1 北極星
オンボーディング完了後 **24時間以内の初回食事ログ完了率** (PRD §10.1)。
Next 期以降は **D7 / D30 リテンション** および **週次アクティブ率** をセカンダリ指標に追加する。

### 1.2 5スイムレーン
| レーン | テーマ | 狙い |
|---|---|---|
| **A. 食事入力** | 入力経路の拡張 (検索 / バーコード / AI) | "記録できなかった" を減らす。`quick_log_unfound_event` の解消 |
| **B. 活動連携** | Health Connect / 動的目標 | TDEE 精度向上、"今日は動いた分食べていい" の納得感 |
| **C. 継続性** | リマインダー / ストリーク / レビュー通知 | 北極星KPIの次層 (D7/D30) を底上げ |
| **D. AI / コーチング** | AI レビュー / 動的ペース提案 / 写真認識 | 入力摩擦の削減 + "次の一手" の提示 |
| **E. 横断** | KPI整備 / マネタイズ / プラットフォーム拡張 | 単発機能ではなく土台寄り、後段の差別化に効く |

### 1.3 ベンチマーク方針 (v1.6)
食事入力レーンは **「あすけん主軸 + 解釈A (最後の砦としての検索)」** で進める (PRD §6.5.0)。DB件数で勝負せず、9ボタン × Identity ツリー × 自動学習というコア体験の軽さを毀損しない。

## 2. Now (MVP リリース直後 〜 4週間) — 計測と即効性のある微調整

> 大型機能を積まない。MVPの素のKPIを4週間で計測し、Next 投入優先度の判断材料を集める期間。

### A. 食事入力
- **直前ログ複製 / お気に入り** (PRD §13.1 P1-A)
  - `expo/utils/quick-log-history.ts` を活用、低コストで高効果
  - ルーチン食事の最短化で初回〜継続体験のフリクション削減
- **brandChips 軽量版投入** (P1-B)
  - 上位コンビニ・チェーン10〜20件のデータのみ先行投入
  - 型 `AmountSpec.brandChips` は既に存在、データ作業のみ
- **あすけん実機ベンチマーク調査** (P1-Y, 必須)
  - 20サンプル (日本食5/コンビニ5/外食5/家庭食材5) でタップ数・結果数・量入力の自然さを計測
  - 結果は SEARCH_SPEC.md と本ドキュメントに反映
  - Next の Identity DB拡張順を決める基礎データ

### B. 活動連携
- 着手しない (Next に集中)

### C. 継続性
- 軽量な空状態コピー / 初回ログ誘導文言の AB を1回転
- 大型施策は Next で

### D. AI / コーチング
- 着手しない

### E. 横断
- **§10.2.1 補助KPI 実装** (必須)
  - `quick_log_unfound_event` / `manual_correction_rate` / `long_press_drilldown_rate`
  - MVP同時計測。Next 投入の判断材料に直結
- **§8.3 Analytics イベント全実装** (必須)
  - Onboarding 完了率・初回ログ完了率の現状値把握
- **学習リセット UI** (PRD §13.1 P2-S9 前倒し検討)
  - 食事傾向の自動学習に違和感が出た場合の安全弁

### Now の判断ルール
4週間で集まる補助KPI + ベンチマーク結果を元に、Next の優先順位を機械的に決定する。

## 3. Next (2〜6ヶ月) — PMF を作る3本柱

> A. 検索系 / B. 活動連携 / C. 継続性 が3本の柱。D, E は補助投資。

### A. 食事入力 (PRD §13.1 P2-S1〜S9)
**検索系を最優先で確定実装** (v1.6 で解釈A方針が定まったため、しきい値判定を待たない):

- **Identity 検索バー再有効化** (P2-S1) — 詳細仕様は [SEARCH_SPEC.md](./SEARCH_SPEC.md)
- **検索精度強化** (P2-S2)
  - ひらがな / カタカナ / ローマ字正規化
  - 同義語辞書 (例: 「ブロッコリー」「ブロッコリ」「broccoli」)
- **Identity DB拡張** (P2-S3)
  - あすけん頻出ベンチマーク準拠で「あすけん頻出 × 自社未収録」を起点に拡張
  - 目標規模: 1,000〜3,000 Identity (DB件数で勝負しない)
- **brandChips 本格投入** (P2-S4)
  - コンビニ / 外食チェーン / 主要パッケージ食品 数百〜千件
- **バーコードスキャン** (P2-S5)
  - Open Food Facts 等の外部DB連携、Identity 紐付けスキーマ拡張
- **検索結果の頻度学習ランキング** (P2-S6) — §6.5.1 自動学習と統合
- **カスタム食品登録** (P2-S7) — 解釈A成立に必須 (DB補完手段)
- **音声入力 → 検索バーへブリッジ** (P2-S8) — 標準 Speech Recognition 利用

### B. 活動連携 (★ ユーザー要望②の本丸)

- **Apple Health / Health Connect 連携 (体重・体脂肪 自動取得)**
  - PRD §6.6.3 既述の「将来検討」を実装フェーズへ
  - WeightEntry / BodyFatEntry スキーマは既に分離済み (PRD §8.2)
- **歩数 / アクティブエネルギーの自動取得**
- **動的TDEE: 活動量による日次kcal目標の補正**
  - 設計論点 (PRD §6.4 更新が必要):
    1. **二重計上の防止**: 既存「活動係数 (1.20〜1.73)」をベースラインとし、当日実績との**差分**のみ動的に上乗せ
    2. **日次キャップ**: ±20% で異常値を吸収
    3. **UX**: ユーザーには「今日 +180 kcal の余裕」のように見せる (kcal目標の置き換えではない)
    4. **PRD更新**: §6.4 算出ロジック / §4.1 In Scope境界 / §10.2 KPI に反映

### C. 継続性

- **週1体重記録リマインダー** (PRD §13 旧P2)
- **食事記録リマインダー** (時間帯学習活用、§6.5.1 の延長)
- **ストリーク / 連続記録バッジ**
  - 中立トーン (PRD §9.1) を厳守。「責めない」設計
  - 例: 「3日連続で記録しています」のように事実ベース、評価語を避ける
- **週次サマリー通知** (Push)
  - /stats 週ビュー (§6.7.3) を push で先回り
- **停滞検知バナー**
  - 体重変化が2週間止まったら My Status にやわらかいヒント
  - 「ペース調整を試してみますか？」程度の中立トーン

### D. AI / コーチング

- **AI 週次レビュー (テキスト)**
  - その週の食事傾向を3〜5行でフィードバック
  - LLM API利用、コスト想定: 週1回 × MAU で見積もり
  - 「責めない・評価しない」トーン (PRD §9.1) を厳守
- **動的ペース調整の提案**
  - 減量停滞時に "標準 → ゆるやか" を提案 (確定はユーザー操作)
  - C レーンの停滞検知バナーと連動

### E. 横断

- **マネタイズ第1弾 (設計フェーズ)**
  - プラン階層構想 (Free / Plus / Pro)
  - トライアル長 / 価格AB の設計
  - Pro 機能ゲーティング (Later で本格運用)
- **Identity ツリー継続拡張** — ユーザーリクエスト経路 (検索結果0件→申請ボタン) を Next 後半で

### Next の判断ルール
- A レーンは確定実装。実機ベンチマーク (Now の P1-Y) 結果でDB拡張順を毎月レビュー
- B レーンの動的TDEE は PRD §6.4 更新を伴うため、設計レビュー必須
- C レーンの中立トーン整合は QA チェック項目に追加
- D レーンの AI 機能は Pro ゲーティング前提で設計

## 4. Later (6〜12ヶ月+) — AI主導 × 長期エンゲージメント × 差別化

### A. 食事入力 (PRD §13.1 P3)

- **写真認識** (P3-A)
  - on-device モデル優先 (Foundation Models on-device or 外部 LLM)
  - 量推定の精度が課題、Pro 限定で試験投入
- **自然言語入力** (P3-B)
  - "鶏むね150gとブロッコリー" → 複数 Identity 分解
- **レシピ分解** (P3-C, 優先度低)
  - 自炊ヘビーユーザーが顕在化したら

### B. 活動連携

- **動的目標フェーズ2**
  - 週単位の活動量から **ペース係数を週次で自動補正** して提案
- **(検討) Apple Watch ペアリング**
  - Quick Log / 体重記録のサブ導線

### C. 継続性

- **長期トレンドビュー**
  - 現在 PRD §4.2 で Out of Scope。Later で再評価
  - 3ヶ月超の体重・体脂肪・体型推移
- **月次レポート (パーソナライズ)**
  - 達成度 + 次月の提案
- **目標達成セレモニー**
  - 体型変化アニメ (現在 → 目標シルエットの遷移)

### D. AI / コーチング

- **動的コーチング**
  - PFC配分の自動微調整提案、停滞時の介入、行動提案
- **対話型コーチ** (Pro)
  - ユーザーの過去ログを文脈にした Q&A

### E. 横断

- **マネタイズ第2弾 (本格運用)**
  - Pro限定機能の整理 (AI写真認識 / 週次レビュー / 対話コーチ)
  - 紹介プログラム
- **プラットフォーム拡張**
  - iPad最適化レイアウト
  - (検討) Web ダッシュボード
- **アクセシビリティ・国際化**
  - i18n、英語対応 (海外ユーザーリーチの選択肢確保)
- **ソーシャル / コミュニティ再評価**
  - PRD §4.2 で Out。価値判断・羞恥を生まない設計が成立するなら長期で再考の余地

## 5. キャパシティと優先順位の前提

### 5.1 想定キャパシティ
- 個人〜小チーム前提 (詳細は別途調整)
- Now: 計測整備 + 軽量改善のみ。新規大型機能は積まない
- Next: 5レーン並行ではなく、A → B → C → D → E の順で重みづけ
- Later: A・D が AI 系で資金 / 投資判断を伴うため、E (マネタイズ) の整備が前提条件

### 5.2 70/20/10 配分
| 配分 | 内容 |
|---|---|
| **70% 計画機能** | 上記の Now / Next / Later 項目 |
| **20% 技術健全性** | 補助KPI整備、Identity ツリー拡張、リファクタリング、パフォーマンス |
| **10% バッファ** | 緊急対応、ユーザー要望の即応、検証用 spike |

## 6. リスクと対応方針

| リスク | 対応 |
|---|---|
| 検索DB拡張が膨張し、コア体験 (9ボタン軽さ) を毀損する | 解釈A方針 (PRD §6.5.0) を四半期レビュー。DB件数の上限を明示的に設ける |
| 動的TDEEで二重計上が発生 (活動係数 + 当日活動量) | Next 着手前に PRD §6.4 を更新し、設計レビューを必須化 |
| ストリーク / バッジが価値判断・羞恥を生む (PRD §9.1 違反) | UX文言レビューを QA チェック項目に追加 |
| AI 機能の API コストが予想を超える | Pro ゲーティング前提で投入、Free 層は月N回制限などで運用検証 |
| ロードマップ変更が頻発する | 月次でレビュー、四半期で正式更新。それ以外は Now の枠内で吸収 |

## 8. 更新ルール

- **月次**: Now の進捗確認、補助KPI 値レビュー、Next 内の優先度微調整
- **四半期**: ロードマップ正式更新、Now → Next → Later の昇格 / 降格判断
- **イベントトリガー更新**: 戦略変更 / 大型ユーザーフィードバック / 競合大型動向時
- **PRD との整合**: ロードマップ変更で PRD §13.1 / §6.5.0 / §10.2 に影響が出る場合は **PRD を先に更新** してから本ドキュメントに反映する
