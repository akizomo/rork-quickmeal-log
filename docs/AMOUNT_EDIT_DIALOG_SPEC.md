# Amount Edit Dialog — Spec

> 量編集ダイアログのユーザーストーリーと受け入れ条件。実装の詳細(寸法・トークン参照・コンポーネントAPI)はコード側の判断に委ねる。
>
> **解決する課題**: 内部テストFB #7「長押しシートで量の入力をするときにキーボードで入力エリアが隠れてしまう」
>
> 関連: [PRD.md](./PRD.md) / [PLAN.md](../PLAN.md) / [QUICK_LOG_DESIGN.md](./QUICK_LOG_DESIGN.md)

---

## 1. 背景と方針

長押しシート(`DishQuickEntrySheet` / `IdentityLogSheet` 等)に量の直接入力欄を埋め込んでいるため、システムキーボード起動時に入力欄・プレビュー・確定ボタンが隠れる問題が発生している。

**方針**: シート側はチップ選択のみで完結させる。量を直接調整したい場合のみ、量表示行のタップで専用ダイアログを開いて編集する。シート側に TextInput を持たないことで、キーボード被り問題を構造的に解消する。

### 利用箇所
長押しシート全般(寿司・ピザ・ご飯・麺・果物・おやつ等)。将来的には既存ログのタップ編集でも同じダイアログを使える想定。

---

## 2. ユーザーストーリー

**US-1**: ユーザーとして、シートに表示されている量がタップ可能であることが見て分かる。なぜなら、編集できると気づかないと辿り着けないから。

**US-2**: ユーザーとして、量表示をタップしたら、その量だけを変更する画面に遷移する。なぜなら、種別選択と量決定は別の操作だから。

**US-3**: ユーザーとして、ダイアログ上で現在値が大きく見える。なぜなら、いま何貫(何g/何切)になっているかを一目で確認したいから。

**US-4**: ユーザーとして、ダイアログ上でステッパー、プリセット、数値入力のいずれかで量を変更できる。なぜなら、微調整したい時とジャンプして変えたい時の両方があるから。

**US-5**: ユーザーとして、ダイアログ内でキーボードが起動しても、確定ボタンが画面から消えない。なぜなら、毎回キーボードを閉じてから確定するのは煩わしいから。

**US-6**: ユーザーとして、確定するとシートに戻り、選んだ値が反映されている。なぜなら、ダイアログは量決定のための一時的な場所だから。

**US-7**: ユーザーとして、間違って開いたら値を変えずに戻せる。なぜなら、誤タップは普通に起きるから。

**US-8**: ユーザーとして、入力可能な範囲を超えた値は、エラーで止められるか、許容範囲に丸められる。なぜなら、明らかに不自然な値が記録されるのは避けたいから。

**US-9**: ユーザー(視覚障害・運動障害)として、ダイアログをスクリーンリーダーやキーボード操作で利用できる。なぜなら、すべてのユーザーが量記録にアクセスできるべきだから。

## 3. 受け入れ条件

### 3.1 トリガー
- [ ] シート内の量表示行が、タップ可能であることを示す視覚的アフォーダンス(編集アイコン・配色・形状のいずれか)を持っている
- [ ] 量表示行をタップするとダイアログが開く
- [ ] 初回利用時に「タップで調整できます」相当の発見性ヒントが何らかの形で示される

### 3.2 ダイアログの構成
- [ ] 現在値が画面上で視覚的に最も強調されている
- [ ] 量を変更する手段としてステッパー・プリセット・数値入力の3つが提供される
- [ ] 値変化時に、関連するプレビュー(例: 概算 kcal)が即時に更新される
- [ ] 確定とキャンセルのボタンが常に同時に存在する

### 3.3 値の操作
- [ ] ステッパーは1ステップごとに値が変化し、最大値・最小値で止まる
- [ ] プリセットを選ぶと、その値で置換される
- [ ] 数値入力に値を打ち込むと、現在値の表示も同期して更新される
- [ ] プリセットに無い値は、どのプリセットも選択状態にならない

### 3.4 確定とキャンセル
- [ ] 確定するとシートに戻り、選択した値がシートに反映される
- [ ] キャンセル/背景タップ/Android 戻るで、値を変えずにシートに戻る
- [ ] 確定済みの値が範囲外の場合は確定できない

### 3.5 キーボード(最重要)
- [ ] 数値入力欄にフォーカスが当たり、システムキーボードが起動しても、確定ボタンが画面外に隠れない
- [ ] キーボード起動中でも現在値の表示が見える
- [ ] キーボードを閉じずに値を確定する手段が提供されている(InputAccessoryView、return キー、または常時可視の確定ボタン)

### 3.6 バリデーション
- [ ] 入力可能な範囲外の値はエラー表示か自動丸めのいずれかで処理される
- [ ] 数値以外の入力(全角数字含む)は受け付けないか正規化される
- [ ] エラー時は何が問題かが文章で示される

### 3.7 アクセシビリティ
- [ ] すべての操作要素のタッチターゲットが最低 44pt
- [ ] スクリーンリーダーで現在値・最小最大・単位が読み上げられる
- [ ] 値変化時に変化後の値がスクリーンリーダーに通知される
- [ ] OS設定の Reduce Motion / Dynamic Type に追従する
- [ ] 主要文字のコントラスト比は WCAG 2.1 AA を満たす

### 3.8 その他
- [ ] カテゴリごとに単位ラベル・許容範囲・刻み・プリセットが切り替わる(寿司=貫、ピザ=切、ご飯=g、麺=玉、いちご=粒 など)
- [ ] 同一ダイアログコンポーネントが複数のシートから再利用できる

## 4. UI イメージ(参考スケッチ)

実装上の正確な寸法・トークン・コンポーネント分割はコード側で決定する。下記は意思共有のための概略のみ。

```
シート側(量行のみ抜粋):
  ▸ 8貫 ✎       約 480 kcal     ← タップで下のダイアログへ

量編集ダイアログ:
  [キャンセル]    量を変更    [完了]
  ─────────────────────────────────
                8 貫
              約 480 kcal

           [ − ]   [ + ]

           よく使う量
        [1貫][6貫][10貫][12貫][16貫]

           数値で入力
        [        8         ]
        ※ 1〜30 貫 の範囲で入力
```

---

## 5. エッジケース

- 入力欄が空文字の状態での確定挙動(無効化 or 直前値で確定 のどちらか)
- 全角数字・先頭ゼロ・小数点複数などの非正規入力の正規化
- ダイアログを連続タップで開く操作の多重起動防止
- ハプティクス未対応端末での fallback
- 横向き表示・タブレット表示
- ダイアログ起動中にシート側の状態が変わるレアケース

具体的な処理方針は実装時にコード側で判断する。

---

## 6. スコープ外

- アプリ内ナンバーパッド(システムキーボード非使用版)
- 単位切替(g ↔ 茶碗 など)
- 写真連携での量推定
- 音声入力
- カスタムプリセットのユーザー定義
- RTL ロケール対応(現状 ja-JP 固定)

---

## 7. 未解決事項(議論ポイント)

1. **プレビュー(kcal等)の表示有無** — ダイアログ側で出すか、シート側に任せるか。出すならどの粒度まで(kcal のみ / PFC まで)
2. **範囲外入力の扱い** — エラーで止めるか、最大/最小に自動丸めるか
3. **発見性ヒントの表示回数** — 何回まで出すか、永続表示するか
4. **InputAccessoryView の優先度** — Phase 1 で iOS のみ実装か、Android 同等実装まで揃えてから出すか
5. **小数の許容** — 麺の 0.5玉 のようなケースを認めるか、整数のみで運用するか

これらは実装着手前に決めるべき設計判断。
