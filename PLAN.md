# MVP P0 一括実装：アプリ紹介・Paywall・オンボ3フェーズ・My Status・Today's Log仕上げ

# 実装する体験の全体像

初回起動時に **アプリ紹介 → 7日無料トライアルのPaywall（スタブ）→ オンボーディング3フェーズ → Home** の流れを通す。完了したユーザーはHomeに直接戻るが、設定ハブの My Status からいつでも目標・体重・食事パーソナライズを編集できる。

## 起動時の振り分け

- 初回起動のユーザーには、アプリ紹介（4画面）を最初に出す
- 「7日間無料で始める」を押すとPaywallに進む
- Paywallで「開始」を押すとオンボーディング、「スキップ」でも一応進める（スタブ運用）
- オンボーディング未完了のユーザーは再起動しても途中のステップから再開
- 全部終わったユーザーは直接Homeに着地

## 新しい画面

### 1. アプリ紹介（4画面・横スワイプ）

- 画面1：最短で記録できる（タップで即記録・長押しで調整）
- 画面2：見るべき数字がすぐ分かる（kcal/PFC/今日の進捗）
- 画面3：自分に合った目標を設定できる
- 画面4：7日間無料トライアル訴求
- ページインジケーター・スキップ・利用規約／プライバシーポリシーへのリンク
- デザイン：既存の柔らかいオフホワイト＋セージグリーンのトーンを踏襲、各画面に軽いイラスト的モチーフ（SVGシルエット・カード・ドット等）

### 2. Paywall

- 7日間無料 → その後月額表示（ダミー価格）・自動更新・解約は設定から の説明
- 「7日間無料で始める」主CTA
- 「購入を復元」「あとで」の補助アクション
- 利用規約／プライバシーポリシー
- 実処理はローカルのみ：トライアル開始日時を保存して「残り◯日」を表示

### 3. オンボーディング Phase 1: 現状整理

- 画面A：身長 / 現在体重 / 身体基準（男性基準・女性基準）
- 画面B：現在体脂肪率（任意）＋ 現在のボディタイプをSVGシルエットから選択（5段階・同一ポーズ・中性的）
- 下部に進行インジケーター、戻る可能
- 入力するたびにローカル保存、離脱→再開に対応

### 4. オンボーディング Phase 2: 目標設定

- 画面C：変わりたい方向（減らしたい・維持・増やしたい）
- 画面D：目標ボディタイプ（Phase1と同じスケール上にシルエットを並べ、現在位置と目標位置が一目で分かる）
- 画面E：おすすめ目標プレビュー（目標体重・目標体脂肪率・1日kcal・PFC）
  - 最初は自動計算された提案値を表示
  - ± で微調整するとkcal/PFCがリアルタイム再計算
  - 整合が崩れたら軽いヒント（「強度の高い目標です」等）

### 5. オンボーディング Phase 3: 食事パーソナライズ

- 画面F：朝・昼・晩・間食ごとの食事スタイル（自炊／外食／コンビニ・中食／まちまち／未設定）
- 画面G：よく食べるもの（最大5個、一皿料理セクションと食材セクションの混在選択OK）
- 画面H：普段の量（少なめ・ふつう・多め）
- 画面I：設定完了サマリー → Homeへ

### 6. My Status（設定ハブ）

- Home上部のアバター／ステータス領域から遷移
- セクション構成：
  - **今日のスナップショット**：体重・直近の変化（グラフは最小限のスパークライン）
  - **現在の状態**：身長 / 体重 / 体脂肪率 / 身体基準 / 現在ボディタイプ（編集可）
  - **目標**：目標方向 / 目標体重 / 目標体脂肪率 / 目標kcal / P・F・C（編集可・再提案ボタン付き）
  - **食事パーソナライズ**：時間帯別スタイル / よく食べるもの / 普段の量
  - **体重を更新する**：数値入力シート（直近平均との差分表示）
  - **サブスクリプション**：現在のプラン・残り日数・購読管理への導線
  - **アプリ情報**：利用規約 / プライバシーポリシー / バージョン
- それぞれの編集は軽いボトムシート or 簡易モーダルで完結

## Today's Logの仕上げ

- 既存のボトムシートを 3段スナップ（12% / 45% / 88%）に整える
- ドラッグで自然にスナップ、スクロールとの競合を抑える
- meal slotの初期タブは、オンボで保存した「時間帯別の食事スタイル」と実ログ傾向の両方で決まる
- 空リストでも、多量リストでも破綻しないレイアウト

## Homeの調整

- 最上部に My Status への導線（現体重→目標体重の距離、トライアル残り日数の小さな表示）
- 日次フィードバック（subheader / body）が、オンボで設定した目標kcalに対して計算される
- ユーザーがオンボでよく食べるものを選んでいれば、Quick Log の候補順にバイアスがかかる

## デザイントーン

- 既存の「静かな日本語ウェルネス」トーンを踏襲
- 柔らかいオフホワイト背景、セージグリーンのアクセント、丸み、低ノイズ
- ボディタイプのSVGは、同一ポーズ・同一構図で細い線・中性的、優劣を感じさせない
- Paywallは「売り込み」ではなく「静かに価値を伝える」トーン

## データまわり（内部的な保存）

- オンボプロフィール（身長・体重・基準・ボディタイプ・目標・食事スタイル・よく食べるもの・量）を新規保存
- トライアル開始日／購読状態（ローカルのみ）
- 既存のFoodLog / WeightEntry / UserProfile は壊さず、足りないフィールドだけ追加
- 古いデータでも開けるよう、未入力は安全な初期値にフォールバック

## 再確認したいポイント（実装前に決められると助かる）

1. **オンボでの目標値の計算式**：体重変化 kg/月、活動レベルの扱い、Mifflin-St Jeor などのベース式で良いか。活動レベルはオンボで聞かない前提（「ふつう」固定）で進めて良い？
2. **ボディタイプの段階数**：男性基準5段階・女性基準5段階（計10種SVG）で良いか、もしくは共通の抽象シルエット5段階で足りるか
3. **よく食べるものの選択肢**：仕様の一皿9カテゴリ＋食材8カテゴリで固定で良いか、それとも個別メニュー（牛丼・カレーライス等）まで降りる？
4. **Paywall価格表示**：ダミーで "¥600/月" 等の固定文言で良いか、それともRevenueCat接続を見据えて完全にプレースホルダーにするか
5. **アプリ起動時の紹介**：一度スキップ or 完了したら二度と出さない方針で良いか（再表示導線は My Status から任意で）

承認いただければ一気に実装します。



# 1. オンボでの目標値計算式

## 結論

**Mifflin-St Jeor をベース式にしてよい**です。  
そのうえで、**活動レベルはオンボでは聞かず、「ふつう」固定**で進めて大丈夫です。

Mifflin-St Jeor は RMR 推定式として広く使われており、NIH/NCBI の Endotext でも成人の推定式として掲載されています。式は、男性基準が `9.99×体重 + 6.25×身長 - 4.92×年齢 + 5`、女性基準が `9.99×体重 + 6.25×身長 - 4.92×年齢 - 161` です [Source](https://www.ncbi.nlm.nih.gov/books/NBK278991/table/diet-treatment-obes.table12est/)

## なぜこれでよいか

このアプリのオンボーディングは、**精密な栄養指導**ではなく、**納得感のある初期提案**が目的です。  
なので、

- 年齢
- 身長
- 体重
- 男性基準 / 女性基準
- 目標方向
- ボディタイプ差分

までで初期提案を出し、**活動量は最初は聞かない**方が体験として軽いです。

活動レベルは聞き始めると、

- デスクワークだけどジム通いはしている
- 平日は低いが休日は高い
- 自分では判断しにくい  
という問題が出て、オンボの負荷が一気に上がります。

---

## 活動レベルの扱い

### 推奨

**内部固定で「ふつう」**にする。  
ただし、その値はUI上では強く見せない。

### 実装推奨

Mifflin-St Jeor の RMR に対して、活動係数は内部的に **固定の normal factor** をかける。  
MVPでは次のどちらかでよいです。

### 推奨案A

**男女基準ごとに固定**

- male_basis: `1.11`
- female_basis: `1.12`

これは NCBI の表にある “Low Active” に近い扱いで、活動量未取得のMVP初期値としては比較的無難です [Source](https://www.ncbi.nlm.nih.gov/books/NBK278991/table/diet-treatment-obes.table12est/)

### 推奨案B

**完全に共通で 1.1〜1.15 の単一係数** 設計上さらに簡単にするならこちらでもよいです。  
ただし、男性基準/女性基準をすでに取るなら、Aの方が自然です。

### 判断

**MVPでは A を推奨**します。

---

## 体重変化 kg/月 の扱い

ここもオンボで直接入力させる必要はありません。  
**内部パラメータとして持つ**だけでよいです。

### 推奨ルール

- 減量: `-1.0 kg/月` を基本
- 維持: `0 kg/月`
- 増量: `+0.5 kg/月` を基本

さらに、**現在ボディタイプと目標ボディタイプの距離**で少しだけ補正してよいです。

### 例

- 1段階差 → 0.5kg/月
- 2段階差 → 1.0kg/月
- 3段階差以上 → 1.5kg/月 上限

ただしMVPでは複雑にしすぎない方がよいので、最初は

- 減量 = -1.0
- 維持 = 0
- 増量 = +0.5 の固定でも十分です。

## 実装指示

初期目標 kcal は以下で算出する。

**TDEE = RMR(Mifflin-St Jeor) × fixed normal activity factor**

そのうえで、

- 減量: TDEE から一定 deficit
- 維持: TDEE 付近
- 増量: TDEE に一定 surplus

を乗せる。  
PFC 比率自体は一般的な範囲から大きく外さず、P 13–20%、F 20–30%、C 50–65% を土台に、目標方向に応じて調整する方針でよいです [Source](https://www.maff.go.jp/j/syouan/keikaku/soukatu/okome_summary/03/health05.html)

# 2. ボディタイプの段階数

## 結論

**男性基準5段階 + 女性基準5段階、計10種SVGを推奨**します。  
共通の抽象シルエット5段階だけでは、今回は足りません。

## 理由

今回は、

- 男性基準 / 女性基準を取る
- 現在ボディタイプを選ぶ
- 目標ボディタイプを選ぶ
- 目標体重 / 目標体脂肪率に連動させる

という設計です。  
このとき、**体脂肪率の目安レンジは男女基準でかなり違う**ため、共通5段階だけだと提案ロジックと見た目の整合が崩れやすいです。一般的な健康的体脂肪率の目安として、男性 10〜19%、女性 20〜29% が示されています [Source](https://www.healthcare.omron.co.jp/resource/guide/fatness/04.html)

## ただし見た目の方針は重要

**10種SVGにするが、表現は抽象寄り**にしてください。

### 方針

- 同じポーズ
- 同じ構図
- 過度な性的差をつけない
- シルエット差は「脂肪感 / 体格感」の差を中心にする
- “理想/非理想” の価値判断に見えないようにする

つまり、 **「男女で別の世界観」ではなく、「同じUI体系の中で基準だけ分かれている」**  
状態にするのがよいです。

## 実装指示

- `male_basis`: stage1〜stage5
- `female_basis`: stage1〜stage5
- current / target とも同じスケール上で表示
- stage 数は5で固定
- v1ではこれ以上増やさない

# 3. よく食べるものの選択肢

## 結論

**一皿9カテゴリ + 食材8カテゴリで固定**でよいです。  
**個別メニュー（牛丼、カレーライス等）まではオンボで降りない**方がよいです。

## 理由

オンボの目的は、**ログしやすい初期状態を作ること**であって、詳細マスタの好みを全部取ることではありません。

個別メニューまで降りると、

- 選択肢が急増する
- 「ここにない」が別の形で増える
- 地域差・店差が強い
- 一皿料理側に寄りすぎる
- Food側との粒度が合わない

という問題が出ます。

一方で、

- 一皿9カテゴリ
- 食材8カテゴリ

なら、

- Dish寄りユーザー
- Food寄りユーザー
- 混在ユーザー

を全部拾いやすく、**初期並び順のパーソナライズ**には十分です。

## 推奨固定選択肢

### 一皿料理

- ごはんもの
- カレー
- 中華麺
- 和麺
- パスタ
- 寿司
- サンド
- ピザ
- 定食・弁当

### 食材・単品

- ご飯
- 肉・魚
- 卵・大豆
- 野菜・サラダ
- 乳製品
- 果物
- プロテイン・飲料
- スープ

## 実装指示

- 最大4〜5個選択
- 一皿 / 食材の混在可
- ここでの回答は **初期候補の仮説** として使う
- 個別メニューへの学習は、実ログ蓄積後に履歴ベースで行う

つまり、 **オンボではカテゴリ止まり、個別メニューは利用後の学習で補う**  
で進めてください。

# 4. Paywall価格表示

## 結論

**"¥600/月" のような固定ダミー文言で出すのは避けるべき**です。  
最終的には **StoreKit / Google Play / RevenueCat から取得した実際のローカライズ価格を表示する前提**で進めてください。

Apple はサブスクの offer details screen に duration と pricing が含まれると明示しています [Source](https://developer.apple.com/app-store/subscriptions/)  
また RevenueCat も、product variables は端末言語・地域に応じた localized value を返し、価格は通貨記号や区切りも含めて適切に整形されると案内しています [Source](https://www.revenuecat.com/docs/tools/paywalls/creating-paywalls/variables)

## 実務判断

### リリース / TestFlight / ベータ

**固定ダミー価格は使わない**  
→ 実商品 metadata から描画する

### 開発初期だけ

RevenueCat や StoreKit 未接続のローカル開発中だけ、  
**DEV専用プレースホルダー** を使うのは可。  
ただし条件は以下です。

- dev build 限定
- beta / release には絶対出さない
- 文言は実価格に見えないようにする  
例: `価格読み込み中`

## 推奨実装

- UI設計は最初から **dynamic price slot** 前提
- 価格表示部は loading / success / error を持つ
- 価格が取れないときに固定金額を出さない
- ベータでは sandbox / test product で実表示を確認

## 最終指示

**価格文字列は固定埋め込みしない。**  
**RevenueCat接続を見据えるのではなく、最初からその前提で組む。**

# 5. アプリ起動時の紹介の再表示方針

## 結論

**一度スキップまたは完了したら、自動では二度と出さない方針でよい**です。  
再表示導線は **My Status か設定導線から任意**で十分です。

## 理由

この紹介は onboarding ではなく、**初回の価値説明と課金導線**です。  
毎回出るとノイズになるし、有料プロダクトとしても邪魔です。

## 実装方針

### 初回

未ログイン / 未紹介完了時のみ表示

### 以後

- 自動再表示しない
- My Status または Settings から
  - アプリの使い方
  - はじめに見る のような再表示導線を置く

### 例外

**紹介コンテンツの major version を上げたときだけ再表示候補にしてよい**です。  
たとえば、

- `introSeenVersion = 1`
- 新しい課金構造や大幅機能変更で `introVersion = 2` なら、再表示フラグを立てる設計はありです。

## 最終指示

MVPではシンプルに、

- 見た or スキップしたら終了
- 手動再表示のみ で進めて問題ありません。

# これで実装仕様として確定してよい内容

## 確定案

### 目標計算

- RMR: Mifflin-St Jeor
- 活動レベル: オンボで聞かない
- 内部係数: `ふつう` 固定
- 体重変化 rate: 目標方向ベースの内部固定値

### ボディタイプ

- 男性基準5段階
- 女性基準5段階
- 計10SVG
- 表現は抽象寄り

### 食事パーソナライズ

- よく食べるもの = 一皿9 + 食材8
- 個別メニューまでは降りない

### Paywall

- 価格は動的表示前提
- 固定ダミー価格は禁止
- DEV専用 placeholder のみ可

### アプリ紹介

- 1回表示
- 以後は自動再表示なし
- My Status から任意再表示
